# naming 技能

## 核心目标

在创建任何 `wiki/` 页面之前，先确定页面类型，再确定文件名。

禁止“先有标题，再临时起文件名”。
禁止同一类对象混用不同命名风格。

## 总原则

文件命名必须满足两个条件：

1. 文件名直接暴露页面类型或对象类型
2. 同类页面使用同一前缀规则

格式统一为：

`类型前缀-核心名称.md`

不得省略前缀。

## 一、`wiki/concepts/` 命名规则

`concepts/` 只允许以下 5 类前缀：

- `概念-`
- `机制-`
- `方法-`
- `框架-`
- `体系-`

示例：

- `概念-Self-Attention.md`
- `机制-多头注意力.md`
- `机制-KV-Cache.md`
- `方法-混合检索.md`
- `框架-Context_Engineering.md`
- `体系-LLM体系.md`

判定规则：

- 回答“它是什么” -> `概念-`
- 回答“它如何起作用” -> `机制-`
- 回答“这件事怎么做” -> `方法-`
- 回答“问题空间如何组织” -> `框架-`
- 回答“知识地图如何分层” -> `体系-`

## 二、`wiki/entities/` 命名规则

`entities/` 必须使用具体对象前缀。

允许前缀：

- `人物-`
- `组织-`
- `工具-`
- `项目-`
- `产品-`
- `模型-`

示例：

- `人物-Karpathy.md`
- `组织-Anthropic.md`
- `工具-Qdrant.md`
- `产品-Claude.md`
- `模型-Gemini.md`

如果对象既可视为产品也可视为模型，优先选当前知识库中的主叙述角色，但同一对象不得双前缀并存。

## 三、`wiki/summaries/` 命名规则

`summaries/` 统一使用：

- `主题-`

示例：

- `主题-Transformer架构原理.md`
- `主题-RAG系统设计.md`
- `主题-Prompt工程实践.md`

规则：

- summary 文件名必须体现“知识主题”
- 不得直接用单篇原文标题
- 不得省略 `主题-` 前缀

## 四、`wiki/synthesis/` 命名规则

`synthesis/` 必须显式标明综合分析类型。

允许前缀：

- `问题-`
- `对比-`
- `策略-`
- `判断-`

示例：

- `对比-RAG-vs-微调-vs-Agent.md`
- `判断-LLM工程全栈知识体系.md`
- `策略-RAG落地路径.md`

判定规则：

- 回答一个问题 -> `问题-`
- 比较多个对象 -> `对比-`
- 给出做法或方案 -> `策略-`
- 输出评价、结论、判断 -> `判断-`

## 五、生成文件名时的执行步骤

每次创建新页面前，严格执行：

1. 先判断页面属于 `concepts / entities / summaries / synthesis` 哪一层
2. 再判断该页面在该层中的具体对象类型
3. 再生成 `前缀-核心名称.md`
4. 再检查是否已存在同义页面、旧命名页面或无前缀页面
5. 只有确认不冲突，才允许写文件

## 六、硬约束

- 禁止无前缀命名
- 禁止同一目录下混用多套命名体系
- 禁止仅凭直觉命名
- 禁止出现 `多头注意力.md` 这类无类型前缀文件名
- 若已有旧文件不符合规范，新增内容时应优先考虑重命名或合并，而不是继续扩散旧命名
