# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 仓库用途

受约束的 Karpathy LLM Wiki / Obsidian 工程。把 `raw/` 中的分散原始资料**编译**为 `wiki/` 中结构稳定、类型清晰的知识页。这是硬约束，不是比喻——不允许简单导入，必须先判断类型再写入。

活跃实例：`llm-wiki-ai/`（AI 大模型技术体系）
模板母版：`llm-wiki-template/`（派生新实例用）

## 技能指令

| 指令 | 触发条件 |
|------|---------|
| `/ingest [路径]` | 用户要摄入/导入/纳入资料到知识库 |
| `/query` | 用户要基于 wiki 提问，先读 `index.md` 定位再回答 |
| `/lint` | 用户要检查知识库健康状态 |
| `/naming` | 新建或重命名任何页面前**必须**先调用 |

详细执行流水线见 `.claude/skills/` 对应 SKILL.md。

## 铁律约束

- `raw/` **只读**，任何操作绝对不写入或修改
- 每次 Ingest / Query 回写 / Lint 后，**必须**更新 `llm-wiki-ai/index.md` 并追加 `llm-wiki-ai/log.md`
- 新建任何页面前，**必须**先调用 `naming` 确定文件名，禁止自由命名
- 新资料优先更新已有页面，而不是无条件新建

## 页面类型与命名规范

**类型判定是第一步，命名是第二步，不能颠倒。**

### concepts/ — 抽象知识对象（5类）

文件名必须带前缀，对应判定问题：

| 前缀 | 判定问题 | 示例 |
|------|---------|------|
| `概念-` | 它是什么 | `概念-Self-Attention.md` |
| `机制-` | 它如何起作用 | `机制-KV-Cache.md` |
| `方法-` | 这件事怎么做 | `方法-混合检索.md` |
| `框架-` | 问题空间如何组织 | `框架-Context_Engineering.md` |
| `体系-` | 知识地图如何分层 | `体系-LLM体系.md` |

判定边界：脱离具体公司/工具后仍成立 → concepts；有明确身份边界 → entities。

概念页**必须**包含这 6 个部分，缺一不可：
> 一句话定义 → 第一性原理 → 核心机制 → 关键权衡 → 与其他概念的关系 → 应用边界

### entities/ — 具体对象（6类）

前缀：`人物-` / `组织-` / `工具-` / `项目-` / `产品-` / `模型-`

### summaries/ — 主题总结

前缀统一用 `主题-`，例：`主题-RAG系统设计.md`

- 一个知识主题对应一页，聚合多个来源
- **禁止**：单篇原文摘要、直接复用原文标题

### synthesis/ — 综合分析

前缀：`问题-` / `对比-` / `策略-` / `判断-`

- 回答一个明确问题，核心是比较/判断/策略/选型
- **禁止**：主题介绍、单篇摘要混入此目录

## 已知命名技术债

当前 `wiki/` 下存在大量不符合前缀规范的旧文件（例：`多头注意力.md`、`摘要-xxx.md`）。新增内容时应优先考虑重命名或合并，不要继续扩散旧命名风格。`synthesis/` 中的 `摘要-xxx.md` 文件属于类型混放，应迁移至对应 `summaries/` 主题页或归档。

## raw/ 目录结构

```
llm-wiki-ai/raw/
├── archive/    ← 归档资料
├── course/     ← 课程材料
├── engineering/← 工程实践资料
├── papers/     ← 论文
└── prompt/     ← 提示工程相关资料
```

## 内部链接格式

统一使用 Obsidian 风格：`[[页面名]]`，来源回指使用相对路径指向 `raw/`。
