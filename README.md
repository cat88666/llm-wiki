# LLM-WIKI

本仓库是一个受约束的 `Karpathy LLM Wiki` / `Obsidian` 工程。目标是把分散的原始资料编译为结构稳定、类型清晰、可持续维护的知识页。

---
## 一、仓库架构

### 1.1 工程目录

```text
dx-LLM/
├── assets/  静态资源文件
├── llm-wiki-ai/   AI知识库实例
├── llm-wiki-template/ 标准模板实例
└── README.md
```

### 1.2 实例目录

```text
llm-wiki-ai/
├── CLAUDE.md
├── index.md
├── log.md
├── raw/
└── wiki/
    ├── concepts/
    ├── entities/
    ├── summaries/
    └── synthesis/
```

### 1.3 目录职责

- `raw/`
  定义：原始资料层。
  规则铁律：只读，禁止修改。

- `wiki/concepts/`
  定义：抽象知识对象层。
  规则类型：
  - 概念：回答“它是什么”的抽象对象
  - 机制：回答“它如何起作用”的内部运行结构
  - 方法：回答“这件事怎么做”的可复用做法
  - 框架：回答“一个问题空间如何组织”的结构化骨架
  - 体系：回答“整片知识地图如何分层”的全景结构

- `wiki/entities/`
  定义：具体对象层。
  规则类型：
  - 人物
  - 组织
  - 工具
  - 项目
  - 产品
  - 模型

- `wiki/summaries/`
  定义：主题总结层。
  规则类型：
  - 主题总结：围绕某个知识主题做聚合总结，不按单篇资料逐篇建页

- `wiki/synthesis/`
  定义：综合判断层。
  规则类型：
  - 问题：围绕一个明确问题组织回答
  - 对比：围绕多个对象做比较分析
  - 策略：围绕目标给出做法或方案
  - 判断：围绕主张、路径或选型给出结论

- `index.md`
  定义：全局索引。
  规则：结构变化后必须更新。

- `log.md`
  定义：操作日志。
  规则：append-only，只追加，不改历史。

## 二、使用方法

### 2.1 `/ingest`

目标：把 `raw/` 资料编译进现有知识结构。
定位：这是知识编译动作，不是简单导入文件。
执行要求：先判断页面类型，再更新已有结构；如需新建页面，再按统一命名规则落盘。


### 2.2 `/query`

目标：基于现有 wiki 回答问题，而不是脱离知识库自由发挥。
定位：这是知识检索与知识回写动作，不是通用问答。
执行要求：先读 `index.md` 定位相关页面，再综合回答；若形成稳定知识，再决定是否回写。


### 2.3 `/lint`

目标：检查结构是否仍然满足工程约束。
定位：这是知识库健康检查动作，不是普通格式扫描。
执行要求：重点检查页面类型边界、索引一致性、链接有效性、来源完整性和命名一致性。
