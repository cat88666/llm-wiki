# dx-LLM

基于 Karpathy `LLM Wiki` 思路实现的本地 Markdown 知识库项目。这个仓库当前包含两部分：

- `llm-wiki-ai/`：已经在持续维护的 AI / LLM 技术知识库实例
- `llm-wiki-template/`：用于复制新知识库的标准模板

项目的核心目标不是“把文章堆进目录”，而是把原始资料持续编译为结构化 wiki：原文放在 `raw/`，知识沉淀到 `wiki/`，具体操作规则放在 `CLAUDE.md`。

---

## 项目现状

当前仓库中真实存在的顶层目录如下：

```text
dx-LLM/
├── assets/
├── llm-wiki-ai/
├── llm-wiki-template/
└── README.md
```

与旧版 README 不同，仓库里目前**没有** `llm-wiki-go/`、`llm-wiki-java/`、`llm-wiki-python/` 等实例。现阶段这个仓库是“一个运行中的实例 + 一个母版模板”的状态，而不是多实例矩阵。

---

## 这个项目解决什么问题

传统知识管理通常只有两种形态：

- 原文堆积：资料越来越多，但难以复用
- 检索式问答：每次重新检索，回答不稳定，也不沉淀知识

这个项目采用更接近“知识编译”的方式：

1. `raw/` 保存原始资料，只读，不让 AI 反复污染原文。
2. `wiki/` 保存 AI 维护后的结构化知识页，分为概念、实体、主题总结和综合分析。
3. `index.md` / `log.md` 负责索引与操作追踪，使知识库可维护、可审计、可迭代。

结果是：新增一篇资料，不只是“多一个文件”，而是会更新已有主题、补充概念关系、形成更稳定的知识结构。

## 仓库结构

### 1. 运行中的实例：`llm-wiki-ai/`

这是当前主要内容库，知识域是 AI 大模型技术体系。

```text
llm-wiki-ai/
├── CLAUDE.md
├── index.md
├── log.md
├── raw/
│   ├── course/
│   ├── engineering/
│   └── prompt/
└── wiki/
    ├── concepts/
    ├── entities/
    ├── summaries/
    └── synthesis/
```

当前实际规模：

- `raw/`：24 篇原始资料
- `wiki/summaries/`：42 页
- `wiki/concepts/`：22 页
- `wiki/entities/`：10 页
- `wiki/synthesis/`：15 页

内容来源主要分为三类：

- `raw/engineering/`：机器学习、Transformer、LLM 原理、RAG、Agent、生产化
- `raw/prompt/`：通用提示工程、Anthropic / Gemini 提示策略、5C 等资料
- `raw/course/`：课程型材料，作为主题补充来源

### 2. 母版模板：`llm-wiki-template/`

模板库用于新建知识库实例，包含：

- `CLAUDE.md`：统一规范
- `wiki/templates/`：四类页面模板
- `demo-*` 示例页面：展示推荐组织方式

适合在新建知识域时直接复制一份，再替换 demo 内容。

## 核心模型

每个实例都围绕三层结构运行：

```text
raw/      原始资料层，只读
wiki/     知识沉淀层，AI 可维护
CLAUDE.md 规则层，定义 Ingest / Query / Lint 行为
```

知识页分工如下：

- `concepts/`：概念、机制、方法
- `entities/`：人物、组织、工具、项目、模型
- `summaries/`：围绕知识主题做聚合总结，不按单篇文章逐个生成
- `synthesis/`：围绕问题、比较、策略、判断做综合分析

这里最关键的设计点是 `summaries/` 与 `synthesis/` 的分离：

- `summary` 负责沉淀“这个主题本身是什么”
- `synthesis` 负责回答“多个主题之间怎么比较、怎么选型、怎么决策”

这比单纯生成一堆摘要更接近真正可复用的知识库。

## 工作流

### Ingest

把新资料放入 `raw/` 后，按规则执行摄入：

```text
请摄入 raw/engineering/05-工程-RAG.md
```

预期结果不是只新增一篇摘要，而是：

- 更新已有主题总结页
- 创建或补充概念页 / 实体页
- 必要时新增综合分析页
- 更新 `index.md`
- 追加 `log.md`

### Query

直接对知识库提问：

```text
RAG 和微调分别适合什么场景？
```

AI 应优先检索现有 wiki 页面回答；如果回答本身形成稳定知识，可以继续回写到 `summaries/` 或 `synthesis/`。

### Lint

对 wiki 做结构化健康检查：

```text
请对 wiki 做一次健康检查
```

重点检查：

- 重复概念
- 孤立页面
- 失效链接
- 无来源结论
- 页面间描述冲突

## 当前知识库画像

从 `llm-wiki-ai/index.md` 可以看出，这个实例已经不是“零散资料收集”，而是一个有明显分层的 AI / LLM 知识体系，覆盖：

- L1 基础原理：机器学习、Transformer、Tokenizer、推理优化、训练并行
- L2 能力模块：Prompt、Embedding、Chunking、RAG、Agent、MCP、Function Calling、记忆与状态管理
- L3 工程系统：知识库问答、Text2SQL、模型路由、可观测性、评估体系、安全与成本控制

同时保留了两类不同粒度的沉淀：

- 面向长期复用的主题页，如 `Transformer架构原理`、`RAG系统设计`
- 面向来源归档的摘要页，如若干课程材料和提示工程外部资料摘要

这说明项目已经从最初的“主题聚合 wiki”进一步扩展成“主题知识 + 来源摘要 + 决策分析”的混合结构。

## 已知结构差异

深看仓库后，有两点需要在 README 里明确写出来：

1. `llm-wiki-ai/` 的 `index.md` 与 `log.md` 位于实例根目录，而 `llm-wiki-template/` 的对应文件位于 `wiki/` 下。
2. 模板规范与运行实例并未完全对齐，说明仓库还处于持续演进中，而不是严格冻结的脚手架状态。

这不是致命问题，但如果后续要批量复制多个实例，最好先统一这两个目录约定，否则会增加自动化维护成本。

## 如何新建一个知识库实例

推荐流程：

1. 复制 `llm-wiki-template/` 为一个新目录，例如 `llm-wiki-go/`
2. 删除所有 `demo-*` 示例内容
3. 根据知识域调整 `CLAUDE.md`
4. 将原始资料放入 `raw/` 的实际分类目录
5. 初始化 `index.md` / `log.md`
6. 从第一批资料开始执行 Ingest

如果后续要把这个仓库发展成多实例知识库集合，建议先把模板与 `llm-wiki-ai/` 的目录约定统一，再开始批量复制。

## 使用建议

- 不要把 `raw/` 当成草稿区。它应该是稳定来源库，只读。
- 不要按“每篇文章一页摘要”来理解 `summaries/`。那会迅速退化成资料堆。
- 对已有主题优先增量更新，而不是频繁新建页面。
- `synthesis/` 应该承担真正有判断力的内容，例如对比、选型、策略，而不是重复 summary。
- `log.md` 要保持 append-only，否则知识库演化历史会丢失。

## 适合谁用

这个项目更适合下面几类场景：

- 需要长期维护某个技术域知识库的个人研究者
- 想把课程、文章、白皮书沉淀成可复用结构的工程团队
- 希望让 AI 参与知识维护，但又不希望直接污染原始资料的人

如果你的需求只是“临时整理几篇文章”，这个结构会偏重；但如果你想构建一个能持续演化的知识系统，这个项目的方向是合理的。
