## 核心结论

**从「写提示词」到「上下文工程」** 的转变，定义了当下使用 AI 的方式。研究表明，结构良好的提示可在基准测试上把大语言模型表现提升约 **20–40%**。但有一个反直觉的发现：对多数任务而言，**150–300 个英文词（或相当信息量）为较优区间**，而到了约 **3,000 token** 起，推理能力就可能开始下滑。

现代推理模型（如 GPT-5、Claude 4、Gemini 3）与上一代模型的用法有本质不同：它们更字面地遵循指令、默认在内部进行推理，且可能被**传统的思维链（CoT）式提示所伤害**。

三大实验室在结构建议上趋于一致：**先写角色/身份，再写指令，然后是示例、上下文，最后才明确输出格式**。

## 每个提示都需要的七大要素

Anthropic、OpenAI、Google 的共识认为，以下七个要素决定输出质量：

| 要素 | 作用 | 示例 |
| --- | --- | --- |
| **角色/人设** | 激活领域相关知识 | 「你是一家财富 500 强公司的高级数据科学家」 |
| **任务背景** | 背景与受众信息 | 「结果将向董事会汇报」 |
| **清晰指令** | 核心请求本身 | 「用 3 条要点概括」 |
| **顺序步骤** | 保证执行顺序 | 编号的操作列表 |
| **少样本示例** | 示范期望格式 | 3–5 个多样且相关的示例 |
| **输出格式** | 期望的回答结构 | 「以 JSON 返回，字段为……」 |
| **约束条件** | 收窄输出空间 | 「200 词以内，避免行话」 |

**Anthropic 的黄金法则：**「把你的提示给一位几乎不了解该任务的同事看。若对方困惑，模型多半也会困惑。」

该原则适用于各类模型。

## 框架对比：Anthropic、OpenAI 与 Google

各家结构建议高度相似，但存在值得注意的模型差异。

### Anthropic 与 Claude 的做法

Claude 4.x **非常字面地**执行指令。早期版本会推断意图、补全模糊请求——现在你必须**明确写出**期望行为。

Anthropic 按「普适有效性从高到低」推荐这些技巧：

1. 清晰、直接
2. 使用示例（多轮少样本提示）
3. 让 Claude「思考」（思维链）
4. 用 XML 标签组织结构
5. 在系统提示中赋予角色
6. 预填回复以引导格式
7. 将复杂提示拆成链式多步
8. 善用长上下文技巧

**要点：**扩展思考（Extended thinking）在**复杂推理上接近满分**，但在**简单查询上得分很低**——会带来不必要开销。Claude Opus 4.5 在关闭扩展思考时，对 **「think」一词特别敏感**——可改用「consider」「believe」「evaluate」等表述。

### OpenAI 与 GPT 系列的结构

GPT-4.1 与 GPT-5 虽有约 1M token 上下文，但仍对**明确、具体**的提示响应最好。**相互矛盾的指令伤害尤其大**——模型会浪费推理 token 去调和冲突。

OpenAI 建议顺序：先角色与目标，再指令（可细分子类：工具调用、格式等），然后是推理步骤、输出格式、示例、上下文，以及收尾指令。

在智能体类任务中，以下 **三项约将 SWE-bench 分数提高 ~20%**：

1. 坚持性（未完成不停止）
2. 工具调用指引
3. 规划类要求

**对推理模型（o 系列、GPT-5）至关重要：**避免思维链式「一步步想」类提示。这些模型默认在内部推理，再要求「step by step」往往多余，甚至可能损害表现。

### Google 与 Gemini 的做法

Gemini 3 **不再像过去那样依赖复杂提示工程**（例如强行思维链）。若你为 Gemini 2.5 写过很重的技巧，可尝试**简化提示**，并配合 `thinking_level: "high"`——模型可能对冗长的「提示工程话术」过度分析。

Google 建议用 XML 风格标签或 Markdown 标题作清晰分隔：**先放上下文与源材料，中间写主任务指令，最后写格式约束**。

**关键参数：**温度建议保持 **1.0**——在推理任务中调低可能引发循环与性能下降。

| 特性 | Claude 4.x | GPT-4.1 / 5 | Gemini 3 |
| --- | --- | --- | --- |
| 指令遵循 | 非常字面 | 精准、手术刀式 | 字面、宜简化 |
| CoT 提示 | 有益（非扩展思考场景） | o 系列宜避免 | 通常不必要 |
| 首选分隔符 | XML 标签 | Markdown / XML | XML / Markdown |
| 上下文窗口 | 200K+ | 1M token | 128K–2M |
| 关键参数 | Extended thinking | reasoning_effort | thinking_level |

## 提示长度研究（可能出乎你意料）

多项 2024–2025 年研究一致表明：**更长并不等于更好**。

Levy 等人的研究发现，大语言模型「在仅约 **3,000 token** 时推理能力就会明显下滑——远早于技术意义上的最大上下文」。TryChroma 关于「上下文腐烂（context rot）」的研究也表明，token 越多，准确回忆往往越差。

### 按任务复杂度的建议长度

| 任务复杂度 | 建议长度 | 适用场景 |
| --- | --- | --- |
| 简单（摘要、问答） | **50–100 词量级** | 翻译、基础问答 |
| 中等（内容创作） | **150–300 词量级** | 写作、分析、专业工作 |
| 复杂（技术文档等） | **300–500 词 + 链式拆分** | 多步工作流、战略分析 |

**关键结论：**一份结构良好的 **16K token 提示 + 检索增强**，在准确性与相关性上**优于 128K token 的「全塞进去」式提示**。精选高信号信息，胜过盲目求全。

### 为何长度会损害表现？

常见原因包括：

- **近因偏好**：Transformer 对靠后的 token 权重更高——长提示里靠前的重要信息易被低估
- **幻觉率**随长度显著上升
- **延迟**：约每 500 token 增加约 25ms 响应时间
- **信号稀释**：模型能识别无关细节，却难以真正「忽略」它们

## 三种情境下的框架选择

并非每个任务都需要复杂提示工程。可按场景选用：

### APE 框架（简单任务）

**A**ction（行动）、**P**urpose（目的）、**E**xpectation（期望）

> 「用三句话为一位很忙的高管概括这篇文章。」

可覆盖约 **80%** 的直白任务。对简单任务堆砌结构，反而可能损害表现——现代模型可能对冗长提示过度分析。

### CO-STAR 框架（内容创作）

**C**ontext（背景）、**O**bjective（目标）、**S**tyle（风格）、**T**one（语气）、**A**udience（受众）、**R**esponse（回应形式）

最适合需要声音、版式与受众的写作任务。示例：

```
# 背景
我正在写一部以禁酒令时期 1920 年代芝加哥为背景的悬疑小说

# 目标
写一段能抓住读者的开篇段落

# 风格
黑色小说，接近雷蒙德·钱德勒

# 语气
氛围感强、带有不祥预感，并带一点冷幽默

# 受众
喜欢经典黑色小说的成年虚构类读者

# 回应形式
150–200 词的开篇段落
```

显式分区可避免模型把请求的不同维度混在一起。

### RISEN 框架（复杂多步）

**R**ole（角色）、**I**nstructions（指令）、**S**teps（步骤）、**E**nd Goal（终局目标）、**N**arrowing（收窄范围）

适用于分析、战略类任务及需要系统化流程的工作流：

```
角色：擅长客户细分的高级数据分析师

指令：分析提供的客户数据集，识别不同细分群体

步骤：
1. 在购买行为中找规律
2. 按相似特征对客户分组
3. 为每个细分起描述性名称
4. 为每个细分给出营销策略建议

终局目标：为营销团队提供可落地的报告以改进定向投放

收窄范围：聚焦购买历史满 6 个月以上的客户，排除异常值
```

「终局目标」让输出对齐业务需求，而非泛泛的学术探索；预算、时间、范围等约束让建议更可立即执行。

## 易被忽略但确实有效的技巧

以下组件影响大却常被低估：

### 输出锚定（Output anchoring）

由你**先写出回复的开头**。例如预填「三个要点是：1.」可引导模型进入你想要的结构。Anthropic 明确推荐，且在主流模型上普遍有效。

### 分隔符与结构标记

《The Prompt Report》（Schulhoff 等，2024）发现：小幅格式调整——重排示例、增加空白、使用 `###`、`===` 或 XML 标签等分隔——可使准确率波动超过 **30%**。

### 上下文放置策略

对长文档，把指令放在提示**末尾**、置于上下文材料之后。可用「基于上文完整文档……」等锚定语，把注意力拉回任务。

### 校准置信度提示

要求模型用 1–10 自评置信度并说明不确定之处，在强依赖准确性的场景更可靠，但实践中很少人用。

### 情绪/受众框架

「用 5 岁孩子能懂的方式解释」与「向投资人解释」——内容相同但框架不同，会激活不同的表达模式与词汇层级。

## 研究实际告诉我们什么

《The Prompt Report》（Schulhoff 等，2024）系统梳理了 **1,565 篇论文**，归纳出 **58 种基于文本的提示技术**。要点包括：

**思维链存在收益递减。**2025 年 6 月沃顿一项研究显示：对推理模型（o3-mini、o4-mini、Gemini Flash 2.5），CoT 收益很小（约 +2.9% 至 +3.1%），却增加约 20–80% 耗时。对非推理模型，提升幅度不一：Claude Sonnet 3.5（+11.7%）、GPT-4o-mini（+4.4%，统计上不显著）。

**少样本与零样本要分情况。**在强推理任务上，零样本 CoT（如「让我们一步步思考」）有时优于少样本——因为模型可自行生成逻辑路径，而不被可能不具代表性的示例束缚。

**2024–2026 研究的共识：**

- 提示工程可在基准上带来约 20–40% 的性能提升
- 示例**选什么**比**有多少个**更重要——多样性很关键
- 更短往往更好（约 150–300 词为常见甜点区）
- 结构（编号列表、要点、分隔符）有助于一致性
- 现代模型常默认「在想」；强行要求「直接给答案」有时反而损害表现

## 行动清单：从这里开始

### 偶尔使用 AI

从 **APE** 入手：行动 + 目的 + 期望即可覆盖多数需求，不必过度复杂化。

### 经常做内容创作

写作类任务用 **CO-STAR**。显式区分风格 / 语气 / 受众，通常优于一团乱麻式提示。

### 搭建自动化

用 **RISEN** 组织提示：写清步骤、终局目标与约束。务必用**即将上线的那款模型**做测试——为 Claude 3.5 优化的提示，在 Claude 4 上可能**反而有害**。

### 所有场景的通用习惯

1. **尽量把提示控制在约 300 词以内**（或等价信息量）
2. 多部分任务用编号步骤
3. 格式要求放在**最后**
4. 新任务类型准备 **3–5 个多样化示例**
5. 在**生产环境将使用的那款模型**上交叉测试提示

## 未来方向：上下文工程

领域已从单纯的「提示工程」成熟为 Anthropic 所称的 **「上下文工程」**——在系统提示、工具、外部数据与消息历史中，**策展最优 token 集合**的学科。

指导原则：**用尽可能少的高信号 token，最大化得到期望结果的概率。**

最好的提示不是最花哨的，而是**用最少必要上下文达成目标**的那一个。
