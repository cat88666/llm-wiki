## 清晰具体的说明

自定义模型行为的一种有效且高效的方法是向模型提供清晰具体的指令。指令可以采用问题、分步任务的形式，也可以很复杂，例如反映用户的体验和思维模式。

### 输入

输入是提示中您希望模型为其提供回答的必需文本。输入可以是模型回答的问题（问题输入）、模型执行的任务（任务输入）、模型操作的实体（实体输入），或模型完成或继续的部分输入（补全输入）。

| **输入类型** | **提示** | **生成的输出** |
| --- | --- | --- |
| 问题 | ``` 有一家花店专门卖干花花束，起什么店名好？请只列出 5 个店名。 ``` | ``` 以下是专营干花的花店店名建议：1. 恒绽花坊 2. 干花拾光 3. 瓣间珍藏 4. 野风低语 5. 花事未央 ``` |
| 任务 | ``` 请用简单列表列出露营必带的 5 样东西，只要物品名称即可。 ``` | ``` 1. **帐篷：**遮风挡雨。2. **睡袋：**保暖舒适。3. **防潮垫：**隔潮缓冲，睡得更稳。4. **头灯或手电：**夜间照明。5. **急救包：**应对小伤小病。 ``` |
| 实体 | ``` 将下列物品归类为 [大, 小]：大象 老鼠 蜗牛 ``` | ``` 分类结果：* **大象：**大 * **老鼠：**小 * **蜗牛：**小 ``` |

#### 部分输入内容补全

生成式语言模型的工作方式类似于高级自动补全工具。当您提供部分内容时，模型可以提供剩余的内容或者它认为是该内容的延续内容来作为回答。如果您添加了任何示例或上下文，模型可以参考这些示例或上下文。

以下示例提供了包含指令和实体输入的提示：

```
对于给定订单，返回 JSON 对象，字段为 cheeseburger、hamburger、fries、drink，值为对应数量。

订单：一个汉堡和一杯饮料。
```
```
{
  "cheeseburger": 0,
  "hamburger": 1,
  "fries": 0,
  "drink": 1
}
```

虽然模型按提示给出了回答，但用自然语言写清规则有时并不容易，模型也要解读很多内容。例如餐馆菜单品类很多。为缩小 JSON 体积，可约定**未点的菜品不出现在 JSON 里**。这时可提供少样本示例，并写出「输出：」前缀，让模型按同样风格补全：

```
有效字段：cheeseburger、hamburger、fries、drink。

订单：给我一个芝士汉堡和薯条
输出：
{
  "cheeseburger": 1,
  "fries": 1
}

订单：我要两个汉堡、一杯饮料和一份薯条。
输出：
{
  "hamburger": 2,
  "drink": 1,
  "fries": 1
}
```

可见：未点的菜品不会出现在 JSON 中（例如第二笔订单未点芝士汉堡，故输出中不含该字段）。

虽然您可以使用提示指定简单 JSON 响应对象的格式，但我们建议您在为响应指定更复杂的 JSON 架构时，使用 Gemini API 的 [结构化输出](https://ai.google.dev/gemini-api/docs/structured-output?hl=zh-cn) 功能。

### 限制条件

指定有关读取提示或生成回答的任何限制条件。您可以告诉模型要做什么和不应该做什么。例如，您可以在提示中指定一个限制条件，以限制摘要长度：

```
用一句话概括下文：

正文：量子计算机利用量子力学现象进行计算，在特定问题上可比传统计算机快指数级。在极小尺度下，物质同时表现出粒子与波动特性，量子计算借助专用硬件利用这一行为。量子器件的工作原理已超出经典物理的适用范围。大规模部署后，量子计算机可能用于网络安全（破解现有加密并协助设计新加密）、气象学（改进预报）等众多领域。然而，当前最先进的量子计算机在很大程度上仍属实验阶段，尚不实用。
```
```
量子计算机借助量子力学现象，能以远超传统计算机的速度完成计算，有望在网络安全、气象学等领域发挥作用，但目前仍多为实验性且尚未实用。
```

```
除非用户明确要求简洁回答，否则所有问题都应详尽、有细节地回答。
```
```

```
```

    * 绝版与稀缺片源：寻找从未数字化发行、或在主流平台暂不可见的影视内容。
    * …（下文略）

```

```
```

## 零样本提示与少样本提示

您可以在提示中添加示例，以向模型示范想要的结果。模型会尝试识别示例中的模式和关系，并在生成回答时应用它们。包含少量示例的提示称为少样本提示，而不提供示例的提示称为零样本提示。少样本提示通常用于规定模型回答的格式、措辞、范围界定或一般模式。使用具体而多样的示例来帮助模型缩小其重点并生成更准确的结果。

我们建议始终在提示中包含少样本示例。不含少样本示例的提示可能效果不大。事实上，如果您的示例足够清晰地显示当前任务，则您可以从提示中移除指令。

以下零样本提示要求模型选择最佳解释。

```
请为问题选出更好的解释：

问题：雪是怎么形成的？
解释1：空气中的水蒸气在大气中凝结成冰晶，冰晶在降落过程中结合、长大成雪花，最终落到地面堆积成雪。
解释2：水蒸气结成冰晶就形成了雪。
答案：
```
```
解释1更好，因为它更完整地描述了过程，包括冰晶如何结合、在降落中长大成雪花等细节。
```

如果您的用例要求模型生成简洁的回答，您可以在提示中添加偏好简洁回答的示例。

以下提示提供了两个偏好较短解释的示例。在回答中，您可以看到这些示例引导模型选择较短的解释（`解释2`），而不是像之前那样较长的解释（`解释1`）。

```
下面是若干「问题—解释—答案」示例，请学习其格式与偏好（更偏好简短解释）：

问题：天空为什么是蓝的？
解释1：由于瑞利散射，较短波长的蓝光比长波长的红光更容易被散射，所以天空看起来是蓝色。
解释2：瑞利散射效应。
答案：解释2

问题：地震的成因是什么？
解释1：地壳中能量的突然释放。
解释2：构造板块突然滑动或断裂，释放能量形成地震波，使地面震动并可能造成破坏。
答案：解释1

请按上述格式回答下一题：

问题：雪是怎么形成的？
解释1：空气中的水蒸气在大气中凝结成冰晶，冰晶在降落过程中结合、长大成雪花，最终落到地面堆积成雪。
解释2：水蒸气结成冰晶就形成了雪。
答案：
```
```
答案：解释2
```

### 最佳示例数量

Gemini 等模型通常可以通过少量示例来学习模式，但您可能需要进行多次实验，以找到可产生最佳结果的提示中提供的示例数量。但是，如果您添加太多示例，模型的回答可能会与示例 [过拟合](https://developers.google.com/machine-learning/glossary?hl=zh-cn#overfitting) 。

### 采用风格一致的内容形式

确保少样本示例的结构和格式相同，以避免回答具有不需要的格式。在提示中添加少样本示例的主要目标之一是向模型展示回答格式。因此，必须确保在所有示例中使用一致的格式，尤其是注意 XML 标记、空格、换行符和示例分隔符。

## 添加上下文

您可以在提示中添加模型解决问题所需的指令和信息，而不是假设模型具备所有必要信息。这些上下文信息可帮助模型了解您要求其执行的操作的限制和详细信息。

以下示例要求模型提供路由器的问题排查指导：

```
我的 Wi‑Fi 断连了该怎么办？Google Wifi 路由器上的指示灯是黄色并缓慢闪烁。
```
```
Google Wifi 路由器黄灯缓慢闪烁通常表示网络连接有问题。可按以下步骤排查：

1. 重启调制解调器与路由器：
   - 拔掉调制解调器电源，等待约 30 秒。
   - 重新插电，等待完全启动。
   - 拔掉 Google Wifi 路由器电源，等待约 30 秒。
   - 重新插电，等待完全启动。
……（下文略）
```

这个回答看起来是通用的问题排查信息，并非特定于路由器或 LED 指示灯的状态。

如需针对特定路由器自定义回答，您可以在提示中添加该路由器的问题排查指南作为上下文，供模型在回答时参考。

```
仅根据下方文本回答问题，回答中只能使用下文提供的内容，不要添加其他信息。
问题：我的 Wi‑Fi 断连了该怎么办？Google Wifi 路由器上的指示灯是黄色并缓慢闪烁。

颜色：黄色快速闪烁
含义：正按住重置键，设备进行出厂重置。
处理：
若持续按住重置键，约 12 秒后灯会变为黄色常亮；变为常亮后即可松开重置键。

颜色：黄色常亮
含义：路由器正在恢复出厂设置。
处理：
此过程最长可达约 10 分钟。完成后设备会自行重启并开始白色脉冲闪烁，表示可以重新设置。

处理：
严重故障。请对路由器执行出厂重置。若仍为红灯，请联系 Wifi 客服支持。
```

## 将提示分解为各个组成部分

对于需要复杂提示的用例，您可以将内容细分为更简单的组成部分，以帮助模型管理这种复杂情况。

1. **细分指令** ：不要在一个提示中包含多条指令，而要为每条指令创建一个提示。您可以根据用户的输入选择要处理的提示。
2. **串联提示** ：对于涉及多个连续步骤的复杂任务，请将每个步骤设为一个提示，并将这些提示按顺序串联在一起。在这一连串的提示中，序列中一个提示的输出将成为下一个提示的输入。序列中最后一个提示的输出将是最终输出。
3. **汇总回答** ：汇总是指对数据的不同部分执行不同的并行任务，并汇总结果以生成最终输出。例如，您可以指示模型对数据的第一部分执行一项操作，对其余数据执行另一项操作并汇总结果。

## 对模型参数进行实验

您向模型发送的每次调用都包含控制模型如何生成回答的参数值。对于不同的参数值，模型会生成不同的结果。因此请尝试不同的参数值，以获得任务的最佳值。不同模型的可用参数可能有所不同。最常见的参数如下：

1. **最大输出词元数**：指定回答中可生成的词元数量上限。词元约为 4 个字符；100 个词元大约对应 60–80 个英文单词（中文则视具体文本而定）。
2. **温度**：温度控制词元采样的随机性。`temperature` 在生成回答时参与采样，并与 `topP`、`topK` 共同作用。较低温度更适合需要更确定、更少发散的回答；较高温度往往更富变化与创意。温度为 0 表示近似贪心：总是倾向选择当前概率最高的词元。
3. **`topK`**：`topK` 改变模型挑选下一个输出词元的方式。若 `topK` 为 1，则等价于在词表中选概率最高的词元（贪心解码）。若 `topK` 为 3，则从概率最高的 3 个候选（再结合温度）中采样下一步。每一步先对概率最高的 `topK` 个词元采样，再经 `topP` 过滤，并结合温度得到最终词元。
4. **`topP`**：`topP` 同样改变挑选输出词元的方式：按概率从高到低累加，直到累计概率达到 `topP`。例如词元 A、B、C 的概率分别为 0.3、0.2、0.1，且 `topP` 为 0.5，则下一步只在 A 或 B 中（结合温度）选择，C 被排除。默认 `topP` 多为 0.95。
5. **`stop_sequences`** ：设置停止序列，告知模型停止生成内容。停止序列可以是任意字符序列。尽量避免使用可能会出现在生成内容中的字符序列。

## 提示迭代策略

提示设计有时需要进行几次迭代，然后才能稳定获得所需的回答。本部分提供了有关在迭代提示时可以尝试的一些做法的指导：

1. **使用不同的措辞** ：即使含义相同，在提示中使用不同的字词或措辞通常也会使模型生成不同的回答。如果您的提示无法产生预期结果，请尝试重新措辞。
	```
	版本 1：
	怎么烤一个派？
	版本 2：
	给我一个派的食谱建议。
	版本 3：
	有什么好吃的派食谱推荐？
	```
2. **改用类似任务** ：如果您无法让模型遵循您的任务指令，请尝试为其提供可实现相同结果的类似任务的指令。
	此提示指示模型使用预定义的类别对一本书进行分类：
	```
	《奥德赛》属于以下哪一类：
	惊悚
	科幻
	神话
	传记
	```
	```
	《奥德赛》属于 **神话** 类。原因如下：
	* **神话：** 讲述希腊神话英雄奥德修斯在特洛伊战争后返乡的历程，包含诸神、怪物与超自然事件等典型神话元素。
	……（下文略）
	```
	回答正确，但模型超出了选项的范围。您还希望模型仅使用其中一个选项（而不是完整的句子）来回答。在这种情况下，您可以将指令改为单选题，并要求模型选择一个选项。
	```
	选择题：以下哪一项最能描述《奥德赛》这本书？
	选项：
	惊悚
	科幻
	神话
	传记
	```
	```
	正确答案是：神话。
	```
3. **更改提示内容的顺序** ：提示中内容的顺序有时会影响回答。请尝试更改内容顺序，看看对回答有何影响。
	```
	版本 1：
	[示例]
	[上下文]
	[输入]
	版本 2：
	[输入]
	[示例]
	[上下文]
	版本 3：
	[示例]
	[输入]
	[上下文]
	```

## 后备响应

后备回答是当提示或回答触发安全过滤条件时模型返回的回答。后备回答的一个示例是“我无法提供帮助，因为我只是一个语言模型”。

如果模型给出后备回答，请尝试提高温度。

## 接地和代码执行

Gemini 能够使用工具来避免在可能生成错误回答的情况下出现幻觉。

[依托 Google 搜索进行接地](https://ai.google.dev/gemini-api/docs/google-search?hl=zh-cn) 可将 Gemini 模型与实时网络内容相关联，当模型可能需要了解冷门或最新事实时，应启用此功能。

Gemini 的 [代码执行工具](https://ai.google.dev/gemini-api/docs/code-execution?hl=zh-cn) 可让模型生成和运行 Python 代码，并且只要模型需要执行任何类型的算术、计数或计算，就应启用该工具。

## Gemini 3

[Gemini 3 模型](https://ai.google.dev/gemini-api/docs/models?hl=zh-cn#gemini-3) 专为高级推理和指令遵从而设计。它们最能理解直接、结构合理且清晰定义了任务和任何限制条件的提示。建议采用以下做法，以充分发挥 Gemini 3 的性能：

### 核心提示原则

- **用词要准确、直接** ：清晰简洁地说明您的目标。避免使用不必要或过于具有说服力的语言。
- **使用一致的结构** ：使用清晰的分隔符分隔提示的不同部分。XML 样式的标记（例如 `<context>` 、 `<task>` ）或 Markdown 标题都有效。选择一种格式，并在单个提示中始终使用该格式。
- **定义参数** ：明确说明任何模棱两可的术语或参数。
- **控制输出详细程度** ：默认情况下，Gemini 3 模型会提供直接而高效的回答。如果您需要更口语化或更详细的回答，必须在指令中明确提出要求。
- **连贯地处理多模态输入** ：使用文本、图片、音频或视频时，将它们视为同类输入。确保您的说明根据需要清楚地提及每种模态。
- **优先处理关键指令** ：将必要的行为限制、角色定义（角色设定）和输出格式要求放在系统指令中或用户提示的最开头。
- **长上下文的结构** ：提供大量上下文（例如文档、代码）时，请先提供所有上下文。将具体说明或问题放在提示的 *末尾* 。
- **锚定上下文** ：在提供大量数据后，使用清晰的过渡短语来连接上下文和您的查询，例如“根据上述信息…”

### Gemini 3 Flash 策略

- **当前日期准确性** ：在系统指令中添加以下子句，以帮助模型注意当前日期是 2026 年：
	```
	对于需要最新信息的时效性用户问题，在通过工具调用构造检索查询时，
	必须遵循系统提供的当前时间（日期与年份）。请记住：今年是 2026 年。
	```
- **知识截点准确性** ：在系统指令中添加以下子句，让模型了解其知识截点：
	```
	你的知识截点为 2025 年 1 月。
	```
- **事实依据性能** ：在系统指令中添加以下子句（可根据需要进行修改），以提高模型根据所提供的上下文生成回答的能力：
	```
	你是一个严格基于「用户上下文」作答的助手，只能使用该上下文中提供的信息。
	回答时**仅可**依赖上下文中直接陈述的事实；**不得**调用自身知识或常识来补全答案。
	不要对已有事实做推测或引申，只按原文如实呈现。你的回答必须与所给文本完全一致、
	可核对，不留猜测或诠释空间。将所给上下文视为真理边界：凡未在上下文中直接出现
	的事实或细节，一律视为**不存在依据**且**不可采信**。若上下文中没有明确写出
	所需答案，你必须说明「信息不可用」。
	```

### 增强推理和规划能力

Gemini 2.5 和 3 系列模型会自动生成内部“思考”文本，以提升推理性能。因此，一般而言，无需在返回的回答中包含模型大纲、计划或详细推理步骤。对于需要大量推理的问题，简单的请求（例如「回答前先认真思考」）可以提高性能，但会消耗额外的思考词元。

如需了解详情，请参阅 [Gemini 思维](https://ai.google.dev/gemini-api/docs/thinking?hl=zh-cn) 文档。

## 智能体工作流

对于深度智能体工作流，通常需要提供具体指令来控制模型如何推理、规划和执行任务。虽然 Gemini 提供了强大的通用性能，但复杂的智能体通常需要您自行权衡计算成本（延迟与词元消耗）与任务准确率。

为智能体设计提示时，请考虑以下可在智能体中引导的行为维度：

### 推理与策略

模型在采取行动之前进行思考和规划的配置。

- **逻辑分解** ：定义了模型必须分析约束条件、前提条件和操作顺序的详尽程度。
- **问题诊断** ：控制在确定原因时分析的深度以及模型对溯因推理的使用。确定模型是应接受最明显的答案，还是探索复杂且可能性较低的解释。
- **信息详尽程度** ：在分析所有可用政策和文档与优先考虑效率和速度之间进行权衡。

### 执行和可靠性

有关智能体如何自主运行和处理障碍的配置。

- **适应性** ：模型对新数据的反应。确定是应严格遵守初始计划，还是在观测结果与假设相矛盾时立即调整。
- **持久性和恢复性** ：模型尝试自行纠正错误的程度。高持久性可提高成功率，但可能导致词元费用增加或出现循环。
- **风险评估** ：用于评估后果的逻辑。明确区分低风险的探索性操作（读取）和高风险的状态更改（写入）。

### 互动和输出

用于配置智能体与用户通信的方式以及结果的格式。

- **模糊性与权限处理** ：定义了模型何时可以进行假设，何时必须暂停执行以向用户寻求澄清或权限。
- **详细程度** ：控制与工具调用一起生成的文字量。此参数用于确定模型是否向用户说明其操作，或在执行期间保持沉默。
- **精确度和完整性** ：输出所需的保真度。指定模型是否必须解决所有极端情况并提供确切的数字，还是可以接受大致估计。

### 系统指令模板

以下系统指令为示例：研究者用于提升模型在智能体类基准上的表现。此类场景中模型需遵守复杂规则并与用户交互。模板引导智能体成为强推理者与规划者，在上述各维度上落实具体行为，并要求在采取任何行动（工具调用或对用户的回复）前先完成规划与推理。

您可以根据自己的具体使用情形限制条件调整此模板。

```
你是一名极强的推理者与规划者。请用下列关键要求来组织你的计划、思考与回应。

在采取任何行动之前（无论是工具调用*还是*对用户的回复），你必须主动、有条理且独立地完成规划与推理，涵盖：

1）逻辑依赖与约束：将拟采取的行动对照以下因素分析；发生冲突时按重要性排序解决：
    1.1）基于政策的规则、强制前置条件与约束。
    1.2）操作顺序：确保当前行动不会阻断后续必要行动。
        1.2.1）用户可能以任意顺序提出操作请求，但你可能需要重排步骤以最大化任务成功完成概率。
    1.3）其他前置条件（所需信息或所需行动）。
    1.4）用户明确给出的约束或偏好。

2）风险评估：该行动会带来什么后果？新状态是否会在后续引发问题？
    2.1）对探索类任务（如搜索），缺少*可选*参数风险**较低**。**优先用已有信息调用工具，而不是先追问用户**，除非你在「规则 1：逻辑依赖」的推理中认定：该可选信息对计划后续步骤必不可少。

3）溯因推理与假设探索：在每一步，为遇到的问题找出最合理、最可能的原因。
    3.1）不要停留在表面或第一眼原因；最可能的原因未必最简单，可能需要更深推理。
    3.2）假设可能需要进一步查证；验证某一假设可能需多步。
    3.3）按可能性优先排序假设，但不要过早丢弃低概率假设；小概率事件仍可能是根因。

4）结果评估与适应性：上一步观察是否要求你调整计划？
    4.1）若初始假设被证伪，应基于已收集信息主动提出新假设。

5）信息可得性：纳入所有适用与可替代的信息来源，包括：
    5.1）可用工具及其能力边界
    5.2）全部政策、规则、清单与约束
    5.3）历史观察与对话记录
    5.4）只能通过询问用户才能获得的信息

6）精确与可核对：确保推理与当前情境高度贴合、极度精确。
    6.1）在引用政策等信息时，用原文摘录来支撑你的论断。

7）完备性：确保需求、约束、可选项与偏好都被充分纳入计划。
    7.1）冲突解决沿用第 1 条的重要性顺序。
    7.2）避免过早下结论：同一情境下可能存在多个相关选项。
        7.2.1）判断某选项是否相关时，综合第 5 条所列全部信息来源推理。
        7.2.2）有时必须询问用户才能判断某信息是否适用；未经核实不得默认「不适用」。
    7.3）复查第 5 条中的适用信息源，确认与当前状态相关者。

8）坚持与耐心：除非上述推理路径已穷尽，否则不要放弃。
    8.1）不要因耗时或用户显露急躁而轻易放弃。
    8.2）坚持必须「聪明」：对*瞬时*错误（例如「请重试」），在**未触及明确重试上限（如最多 x 次）**前*必须*重试；一旦达到上限*必须*停止。对*其他*错误，必须调整策略或参数，不得机械重复同一失败调用。

9）抑制冲动输出：完成以上全部推理后再行动。行动一旦执行便无法撤回。
```
