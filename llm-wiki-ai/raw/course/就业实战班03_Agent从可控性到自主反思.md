## 二、Agent核心理念与设计哲学

Agent的核心是基于大语言模型（LLM）构建的智能体，LLM充当Agent的“大脑”，负责推理和决策。Agent设计的关键在于如何确定给予LLM的自由度大小，以平衡智能发挥与结果可控性。

- **自由度小**：通过严格的步骤和工作流控制，保证结果准确，减少幻觉和错误，适合对准确性要求高的任务。
- **自由度大**：适用于复杂、难以预设步骤的任务，允许LLM自主推理和行动，但需引入反馈和反思机制以保证结果质量。

Agent与LLM的交互本质是提示词（Prompt）的拼接和管理。提示词模板定义了任务目标、工具调用规则和推理逻辑，是实现智能决策的基础。

## 三、Agent类型与技术细节

### 3.1 Workflow Agent（工作流型）

#### 定义与特点

Workflow Agent基于预定义的步骤序列，通过严格的工作流控制任务执行流程，确保结果的准确性和可控性。适合步骤明确、对准确性要求极高的场景。

#### 设计动因

- 大语言模型存在幻觉风险，且“只会说不会做”，需要人为定义明确步骤。
- 某些任务必须保证结果准确，不能依赖模型自由发挥。

#### 结构组成

| 组件           | 作用说明                                   |
|----------------|--------------------------------------------|
| 大语言模型     | 作为决策大脑，负责调用工具和生成代码       |
| 工作流         | 预定义的步骤序列，控制任务执行流程         |
| 工具/插件/Skill | 具体功能模块，如地图API、代码执行器等      |
| 知识库         | 提供基础信息支持，减少模型幻觉             |
| 提示词模板     | 规范模型推理和决策逻辑                     |
| 数据库/触发器  | 存储数据，支持定时或事件驱动操作           |

#### 设计要点与技巧

- 利用LLM的工具调用能力（Function Call），自动判断是否调用工具及参数。
- 避免让LLM直接做数学计算，改由生成代码交由专门模块执行。
- 结合知识库与工具，提升回答准确性和可控性。
- 工作流模块化设计，方便复用和维护。

#### 案例分析：招聘站点查询Agent

用户询问“某地点附近是否有招聘站点”，知识库无法准确回答具体地理位置相关问题。解决方案包括：

- 利用地理坐标（经纬度）计算距离，筛选附近站点。
- 调用百度地图API获取地点经纬度。
- 由LLM生成Python代码计算距离，代码由执行器运行。
- 最终由LLM整理结果，反馈给用户。

### 3.2 React Agent（推理-行动-反馈循环型）

#### 定义与特点

React Agent基于推理（Reasoning）与行动（Acting）的循环过程，包含推理、行动和反馈整理三个步骤，循环迭代直至任务完成。适合步骤难以预设、任务复杂多变的场景。

#### 设计动因

- 任务复杂，步骤难以预设。
- 需要LLM具备较大自由度，能够根据反馈动态调整策略。
- 模拟人类思考、行动、总结的闭环过程。

#### 关键概念澄清

- **Reasoning（推理）**：LLM的思考和决策过程。
- **Inference（推理）**：模型在推断阶段的数学计算过程，二者需区分。

#### 设计要点

- 限制LLM每轮只能选择一个动作，动作集合由设计者定义，防止无序操作。
- 通过详细提示词模板引导LLM推理、拆解概念、做决策。
- 反馈机制将执行结果结构化反馈给LLM，保证上下文连续性。
- 多轮推理-行动-反馈循环完成复杂任务。

#### 案例分析：基于文件的供应商销售分析Agent

任务为找出“九月份销售额不达标的供应商”，涉及多个文件（PDF、Excel）包含销售标准、联系人信息、销售数据。挑战在于文件多且结构复杂，难以预设操作步骤。

实现流程：

- LLM通过提示词模板引导，拆解任务，规划动作。
- 设计5个可选动作：列出文件名、查询文档、查看Excel结构、深度分析Excel、结束任务。
- 每轮循环LLM选择一个动作输出指令。
- 预先开发的代码或工具执行动作，返回结果。
- 结果反馈给LLM，更新提示词，进入下一轮推理。
- 多轮循环后，LLM综合信息给出最终答案。

## 四、语言模型版本对比与行为分析

### 4.1 模型版本表现差异

以“千问三”与“千问三点五”为例，展示不同版本语言模型在Agent任务执行中的表现差异：

- **千问三**版本表现更稳定，推理过程合理，能够较好完成任务。

### 4.2 具体案例分析

在“9月份销售额数据提取”任务中，模型需从多个文件中选择合适数据源：

- 模型通过文件名联想，推断“9月份销售记录”可能存储于Excel文件。
- 推测“达标标准”可能存在于PDF文件，体现一定推理能力和上下文理解。
- 模型选择了“深度分析Excel文件”动作，生成了相应代码成功读取Excel前三行数据。
- 虽选择合理，但整体任务角度看未必最优，反映模型决策阶段的局限。

### 4.3 Agent工作流程与职责分工

Agent任务执行分为多阶段：

1. **推理阶段**：语言模型根据任务描述和上下文，输出决策和行动方案。
2. **执行阶段**：预开发代码模块根据模型指令执行具体操作，如读取Excel、解析PDF。

语言模型负责决策制定，代码模块负责具体执行，确保系统灵活性和可控性，但模型决策准确性直接影响任务完成效果。

## 五、可控性与自主反思的探讨

### 5.1 决策的无对错性

在某些步骤中，模型选择无绝对对错，例如先查看Excel还是PDF文件均有合理性，体现Agent多路径决策的灵活性，但也带来决策不确定性。

### 5.2 自主反思的必要性

模型可能出现死循环或错误选择，故引入自主反思机制尤为重要。自主反思帮助模型识别并纠正错误决策，提升任务完成的准确率和效率。
