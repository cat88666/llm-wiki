# 大模型 LLMs 原理

## 一、Tokenizer

### 1. 为什么 Tokenizer 很重要

Tokenizer 决定了文本如何被切成 token，它直接影响：

- 上下文窗口利用率
- 训练和推理成本
- 中英文、多语言表现
- OOV 处理能力
- 语义切分粒度

不要把 Tokenizer 当成“预处理小问题”。它会影响 embedding、上下文长度、RAG 切片效果和推理成本。

### 2. BPE

Byte-Pair Encoding 的核心思路：

1. 准备训练语料和目标词表大小
2. 先把词拆成字符级或字节级单位
3. 统计高频相邻子词对
4. 每轮合并最常见的相邻对
5. 重复直到达到目标词表大小或继续合并已经没有收益

特点：

- 简单直接
- 高频子词容易保留
- 对未登录词有较好适应能力
- 容易出现中文切得过碎的问题

常见采用者：GPT-2、BART、LLaMA 系列。

### 3. WordPiece

WordPiece 本质上和 BPE 很接近，但合并准则不同。

- BPE 更偏向“出现频率最高的相邻子词”
- WordPiece 更偏向“能最大化语言模型概率提升的合并方式”

特点：

- 更偏语言模型导向
- 通常分词效果更稳
- 训练过程相对更依赖概率建模

常见采用者：BERT 系列。

### 4. SentencePiece

SentencePiece 的核心特点是把空格也当作普通符号处理，不强依赖外部分词器。

优点：

- 适合多语言
- 对无空格语言更友好
- 训练和部署更统一

它既可以基于 BPE，也可以基于 Unigram Language Model。

常见采用者：ChatGLM、BLOOM、PaLM。

### 5. 不同大模型分词方式的差异

面试可从 4 个角度比较：

- 词表大小
- 中文切分粒度
- 英文编码效率
- 多语言兼容性

典型结论：

- LLaMA 词表相对较小，中文 token 往往更碎，平均 token 数更高
- 扩展中文词表后，中文编码效率会明显提升
- ChatGLM 的中英文平衡通常更好
- BLOOM 虽然词表大，但由于要覆盖多语种，不一定在单语种上极致优化

面试表达建议：

“Tokenizer 设计是词表大小、分词粒度、多语言泛化与编码效率之间的平衡问题。”

## 二、分布式训练与并行策略

### 1. 单卡训练的现实边界

如果只用一张卡训练或微调 LLM，首先受限于显存。

经验上：

- 小模型微调还能勉强单卡做
- 一旦参数量上去，除了模型参数本身，还要考虑梯度、优化器状态、激活值
- 显存不够时，训练往往比推理先崩

常见经验判断：

- 推理只需要参数和 KV Cache
- 训练还要额外存梯度、优化器状态、激活
- 同样的模型，训练显存远高于推理显存

### 2. 三类核心并行

#### 2.1 数据并行 DP

每张卡持有完整模型，处理不同 batch，再同步梯度。

优点：

- 最容易实现
- 对模型结构侵入小

缺点：

- 单卡必须放得下完整模型
- 卡数增多后梯度同步通信开销会上升

适合：

- 模型能装进单卡
- 多卡加速训练吞吐

#### 2.2 张量并行 TP

把同一层的计算拆到多张卡上。

优点：

- 可解决单卡放不下的问题
- 对超大模型很关键

缺点：

- 通信要求很高
- 通常要求节点内 NVLink / NVSwitch

适合：

- 节点内高速互联
- 模型太大无法单卡完整容纳

#### 2.3 流水线并行 PP

按层切分模型，不同卡负责不同层。

优点：

- 直观地解决模型装不下问题
- 对通信要求低于 TP

缺点：

- 有流水气泡问题
- 推理时容易出现串行瓶颈

适合：

- 没有高速互联
- 穷人多卡方案

### 3. 3D 并行

DP + TP + PP 叠加，就是常说的 3D 并行。

适用场景：

- 超大规模预训练
- 单一并行策略无法同时满足显存和吞吐要求

核心认识：

- TP 更吃通信
- PP 更容易落地
- DP 主要扩吞吐
- 真正的大模型预训练一般要多策略组合

### 4. ZeRO 与 FSDP

ZeRO 的核心是把参数、梯度、优化器状态分片，减少数据并行中的冗余存储。

常见理解：

- ZeRO-1：分片优化器状态
- ZeRO-2：再分片梯度
- ZeRO-3：进一步分片参数

FSDP 可以理解为 PyTorch 体系下与 ZeRO-3 思路相近的全分片方案。

优点：

- 显存节省明显
- 对超大模型训练很有帮助

缺点：

- 通信压力大
- 对网络和工程调优要求高

### 5. 为什么有 ZeRO 还需要 3D 并行

这是典型面试题。

回答思路：

- ZeRO 解决的是数据并行冗余问题
- TP 解决的是单层计算与模型切分问题
- PP 解决的是层级切分问题
- 当模型规模极大时，单靠 ZeRO 仍可能不够

进一步说：

- ZeRO 在跨节点慢网络环境下很容易吃通信亏
- TP 虽然通信大，但在节点内高速互联下非常有效
- 真正超大规模训练通常需要 ZeRO/FSDP 与 TP/PP 组合

### 6. 并行策略如何选

可按场景记忆：

#### 6.1 单 GPU

- 显存够：直接训
- 显存不够：offload 到 CPU，但速度会变慢

#### 6.2 单节点多卡

- 单卡能装下模型：DDP 或 ZeRO
- 单卡装不下：TP、PP 或 ZeRO
- 没有 NVLink 时，优先考虑 PP

#### 6.3 多节点多卡

- 网络快：ZeRO 或 3D 并行
- 网络慢：尽量避免重通信方案
- 显存少又网络慢时，设计会非常痛苦

### 7. 大规模训练的工程经验

原文里有几个很实用的点：

- 多机训练速度不如单机，很多时候不是算力问题，而是网络带宽打满
- TP 适合节点内高速互联，不适合跨慢网
- 万兆网跑超大模型 ZeRO-3，通信几乎会拖垮训练
- 大规模集群下 GPU 数量越多，单卡利用率不一定越高

面试表达可以这样说：

“分布式训练的关键不是把任务切出去，而是显存、吞吐和通信成本的整体平衡。”

### 8. 常见工程问题

#### 8.1 多机找不到主机

排查点：

- hostfile 用 IP，不用 hostname
- 各节点 SSH、端口和网卡配置一致

#### 8.2 NCCL 不通

常见思路：

- 配 `.deepspeed_env`
- 指定 `NCCL_SOCKET_IFNAME`
- 必要时关闭 `IB`、`P2P`
- 打开 `NCCL_DEBUG=INFO`

#### 8.3 多机比单机慢

核心排查：

- 带宽是否打满
- 通信拓扑是否合理
- batch size 是否过小
- TP/PP/DP 组合是否匹配硬件

## 三、Normalization 篇

### 1. LayerNorm

LayerNorm 对每个样本的隐藏维度做归一化，而不是对 batch 维度做归一化。

优点：

- 不依赖 batch 统计量
- 更适合变长序列和 Transformer
- 训练稳定性好

面试重点：

- 为什么 Transformer 更偏好 LayerNorm 而不是 BatchNorm
- 因为序列建模时 batch 统计不稳定，且不同位置之间计算方式与 CNN 不同

### 2. RMSNorm

RMSNorm 可以看作 LayerNorm 的简化版。

关键区别：

- 去掉了减均值这一步
- 只基于均方根做缩放

优点：

- 计算更简洁
- 速度更快
- 在很多 LLM 中效果与 LN 接近，甚至略优

### 3. DeepNorm

DeepNorm 的核心思想：

- 在残差路径上做放大
- 在初始化阶段对参数做缩放

目标：

- 控制深层网络中的更新幅度
- 提高超深 Transformer 的训练稳定性

面试回答关键词：

- up-scale residual
- down-scale initialization
- 限制爆炸式更新

### 4. LN 放置位置

#### 4.1 Post-LN

- LayerNorm 放在残差之后
- 早期 Transformer 常见

问题：

- 深层网络训练更不稳定
- 梯度更容易放大

#### 4.2 Pre-LN

- LayerNorm 放在子层前

优点：

- 深层训练更稳定
- 梯度传播更顺

缺点：

- 某些设置下效果略逊于 Post-LN

#### 4.3 Sandwich-LN

- 在 Pre-LN 基础上插入额外 LN

用途：

- 某些工作中用于缓解数值爆炸

风险：

- 结构更复杂
- 训练调参难度更高

### 5. 归一化面试结论

- LN 是 Transformer 默认选项
- RMSNorm 是高效简化版本
- DeepNorm 更偏向超深模型稳定训练
- Pre-LN 更适合现代大模型

## 四、激活函数与 FFN 篇

### 1. FFN 的作用

Transformer 里的 FFN 不只是“线性层堆叠”，它负责：

- 通道维度上的非线性变换
- 表征能力增强
- 与 Attention 形成互补

经典结构通常是：

`Linear -> Activation -> Linear`

### 2. 常见激活函数

#### 2.1 GeLU

特点：

- 平滑
- 对小输入保留概率性激活
- 在 Transformer 中非常常见

#### 2.2 Swish / SiLU

特点：

- 连续可导
- 实践中常有更好表现

#### 2.3 GLU 系列

把激活和门控结合起来，常见变体包括：

- GLU
- GeGLU
- SwiGLU

其中 SwiGLU 在很多新一代 LLM 中更常见。

### 3. 为什么 GLU 系列受欢迎

因为它相当于引入了显式门控机制：

- 能提升表达能力
- 训练更稳定
- 在相同参数预算下常优于传统 FFN

代价：

- 结构稍复杂
- 参数组织方式不同

### 4. 面试可答的结论

- 传统 FFN 常配 GeLU
- 新一代大模型更常见 SwiGLU / GeGLU
- 选择激活函数本质是在表达能力、训练稳定性和效率之间权衡

## 五、Attention 优化篇

### 1. 传统 Attention 的问题

最核心的问题有两个：

- 时间复杂度高
- 显存占用大

原因是标准自注意力要构造 `n x n` 相关性矩阵，序列越长，开销越高。

### 2. Attention 优化方向

原文涉及的优化方向可以系统化为：

- 延长上下文长度
- 降低时间复杂度
- 降低显存与 KV Cache 开销
- 优化 kernel 和内存访问
- 改进多头机制

### 3. 常见变体

常见类别包括：

- 稀疏 Attention
- 线性 Attention
- 原型 / 内存压缩类 Attention
- 低秩或低阶近似
- 引入先验的 Attention
- 多头机制改造

面试不用把所有论文背下来，但要知道它们都在解决“长序列高开销”问题。

### 4. Multi-Query Attention

#### 4.1 背景

标准 MHA 中，每个头都有自己的 K 和 V。推理阶段随着序列增长，KV Cache 很大，容易成为瓶颈。

#### 4.2 核心思路

MQA 让所有头共享 K 和 V，只保留多头 Q。

#### 4.3 优点

- 大幅减少 KV Cache
- 降低显存占用
- 提升推理速度

#### 4.4 代价

- 表达能力可能有轻微损失
- 部分模型需要通过调整隐藏维度或 FFN 配置来补偿

代表模型：PaLM、Falcon、ChatGLM2。

### 5. Grouped-Query Attention

GQA 是介于 MHA 和 MQA 之间的折中。

做法：

- 不是所有头完全共享 K/V
- 而是多个 Query 头共享一组 K/V

优点：

- 比 MHA 更省缓存
- 比 MQA 更保留表达能力

代表模型：LLaMA2 34B/70B、ChatGLM2。

### 6. FlashAttention

这是高频面试题。

核心不是“改了模型定义”，而是：

- 通过 kernel 融合、分块计算、在线 softmax、重计算等方式
- 减少 HBM 读写
- 更高效利用 SRAM

效果：

- 更省显存
- 更快
- 尤其适合长序列训练和推理

关键词必须会说：

- HBM
- SRAM
- tile / block
- online softmax
- kernel fusion

### 7. 并行 Transformer Block

有些模型会把原本串行的模块改成并行结构，提升训练吞吐。

收益：

- 训练提速

代价：

- 小模型可能有轻微效果损失
- 大模型上通常影响较小

## 六、损失函数篇

### 1. KL 散度

KL 散度用于衡量两个概率分布之间的差异。

特点：

- 非对称
- 值非负
- 越小说明两个分布越接近

在大模型里常见于：

- 蒸馏
- 分布匹配
- RLHF / DPO 等约束项

### 2. 交叉熵

交叉熵本质上也是在衡量预测分布和真实分布差异。

分类任务中最常用，因为：

- 与概率输出天然契合
- 梯度性质好
- 对错误高置信预测惩罚更强

### 3. KL 与交叉熵的关系

可以这样答：

- 交叉熵 = 熵 + KL 散度中的偏移项
- 当真实分布固定时，最小化交叉熵等价于最小化 KL

### 4. 为什么分类用交叉熵而不是 MSE

面试标准答案：

- 分类输出是概率分布
- 交叉熵对概率差异更敏感
- Softmax / Sigmoid 配合交叉熵，优化更稳定
- MSE 更适合回归，不适合概率空间分类

### 5. 多任务 loss 差异过大怎么办

原文给出的思路可以整理为：

- 动态调权
- 任务特定 loss
- 调整模型结构
- 正则化约束

更工程化一点的回答可以补充：

- uncertainty weighting
- GradNorm
- loss clipping
- 分阶段训练

### 6. Softmax 数值溢出怎么办

标准答法：

- 在计算 `exp(x)` 前先减去最大值

也就是：

`softmax(x) = exp(x - max(x)) / sum(exp(x - max(x)))`

### 7. 信息增益

虽然这是传统机器学习概念，但面试仍会问。

本质：

- 信息增益衡量一个特征让样本不确定性下降了多少
- 决策树用它选择分裂特征

## 七、相似度与对比学习篇

### 1. 常见相似度方法

除了余弦相似度，还常见：

- 欧氏距离
- 曼哈顿距离
- Jaccard 相似度
- 皮尔逊相关系数
- 点积

在 embedding 检索里，最常见还是：

- cosine similarity
- inner product
- L2 distance

### 2. 对比学习是什么

对比学习的目标是：

- 拉近相似样本表示
- 拉远不相似样本表示

常见形式：

- Siamese
- Triplet Loss
- InfoNCE

在大模型领域，对比学习常用于：

- embedding 模型训练
- 检索模型训练
- 多模态对齐

### 3. 负样本为什么重要

负样本决定了表示空间的分离能力。

如果负样本构造成本高，可用的思路有：

- 更高效的采样策略
- in-batch negatives
- hard negative mining
- 利用预训练模型生成伪负样本
- 借助迁移学习减少人工构造成本

## 八、LLM 训练经验篇

### 1. 训练框架怎么选

原文的经验倾向很明确：

- 节点少时，框架差异不一定是核心矛盾
- 节点多时，DeepSpeed 在启动、分析和分布式能力上更有优势

更完整的回答：

- 小规模实验：Transformers + torchrun 够用
- 大规模训练：DeepSpeed / Megatron-LM / FSDP 更常见
- 真正超大规模：往往是多框架混合

### 2. 训练前必须想清楚的事

- 训练目标是什么
- 为什么不是直接用现成模型
- 是继续预训练、SFT、偏好对齐，还是只做 RAG
- 成本预算是多少
- 成功指标是什么

### 3. 稳定训练的经验

- 开启弹性容错
- 自动重启
- 定期 checkpoint
- 保留日志和配置
- 记录每次实验差异

面试时可以直接说：

“大模型训练不是单次跑通，而是长期稳定地跑完。”

### 4. 关注真实 GPU 利用率

不能只看 `nvidia-smi`。

更关键的是：

- TFLOPS
- 吞吐率
- step time
- data loading 比例
- 通信占比

### 5. 环境问题

原文强调了一个非常现实的问题：

- Python
- pip
- virtualenv
- setuptools
- CUDA / NCCL
- GLIBC

这些版本不一致，分布式训练极易翻车。

建议：

- 优先 Docker
- 底层库别轻易升级

### 6. 模型大小如何选

建议从小模型先跑通，再扩到大模型。

原因：

- 便于排查
- 成本低
- 可以先验证流程、数据和损失设计

### 7. 加速卡如何选

原文倾向很明确：

- 时间紧、要稳定，优先 NVIDIA
- 异构硬件或国产卡要预留更多适配成本

## 九、RAG 基础篇

### 1. 为什么需要 RAG

直接把外部知识“训进模型”并不总是最优方案。

原因：

- 微调成本高
- 知识更新慢
- 数据构造成本大
- 很难保证全部知识都被参数记住

RAG 的本质是：

- 把检索系统接到生成系统前面
- 用外部知识增强回答

### 2. RAG 标准流程

1. 加载文档
2. 文本清洗
3. 切分 chunk
4. chunk 向量化
5. 用户 query 向量化
6. 相似度检索 top-k
7. 将上下文和问题拼入 prompt
8. 调用 LLM 生成答案

### 3. RAG 核心技术

原文强调的核心是 embedding。

因为 embedding 决定了：

- query 和文档是否能在同一语义空间对齐
- 检索召回是否靠谱
- 多语言和垂直领域效果是否可用

### 4. RAG Prompt 模板

实用模板通常包括：

- 已知上下文
- 回答规则
- 无法回答时的拒答策略
- 输出语言或格式要求

例如：

1. 只根据提供的上下文回答
2. 如果信息不足，明确说无法回答
3. 不允许编造
4. 指定输出语言和风格

## 十、RAG 优化篇

### 1. 最大痛点：文档切分

原文对这个问题讨论很多，核心观点是对的：

- chunk 太小，语义不完整
- chunk 太大，噪声太多
- 简单按换行或固定长度切分，不适合复杂 PDF 和跨段落问题

因此，文档切分最好是语义驱动，而不是纯字符驱动。

### 2. 语义切分思路

可整理为两类：

#### 2.1 基于篇章结构

- 利用 discourse parsing
- 合并有主从、承接关系的段落

目标：

- 保证一个 chunk 尽量只讨论一件事

#### 2.2 基于语义相似

- 用 BERT / embedding 模型判断相邻句段是否衔接
- 超过阈值则合并，否则切断

这对中文长文档尤其有用。

### 3. 二级索引思想

原文里一个很重要的工程思路是“二级索引”：

- 一级索引：关键信息
- 二级索引：原始文本

检索时：

- 对关键信息做 embedding
- 召回后映射回原始文本

优点：

- 检索更准
- 噪声更少
- 上下文仍能保持完整

### 4. 关键信息怎么抽

原文给了几类办法：

#### 4.1 constituency parsing + NER

抽取：

- 名词短语
- 动词短语
- 实体名

适合结构化较强文本。

#### 4.2 语义角色标注 SRL

抽取“谁对谁做了什么”。

适合强调动作关系和事件结构。

#### 4.3 关键词抽取

工具包括：

- HanLP
- KeyBERT

但要注意：

- HanLP 中文效果好，商业化较多
- KeyBERT 英文更友好，中文未必稳

#### 4.4 垂直领域专用模型

在法律、金融、医疗等领域，通用关键词工具可能不够，直接训练 KeyLLM 或专用关键词生成器更有效。

### 5. 常见 RAG 痛点与对策

#### 5.1 垂直领域效果差

原因：

- embedding 不适配领域
- LLM 不懂领域术语

对策：

- 领域 embedding 微调
- LLM 领域数据微调

#### 5.2 LangChain 默认问答效果差

可能原因：

- 文档切分粗糙
- 填充上下文时噪声太大
- 默认检索器不适合业务

对策：

- 使用更好的分段方式
- 只补充高相关上下文
- 对每段先做总结，再做匹配

#### 5.3 召回率低

应从三点同时看：

- Document 长度
- embedding 质量
- top-k 大小

常见对策：

- 更高质量 embedding
- 更短 chunk + 更大 top-k
- 向量检索与 ES / BM25 混合召回
- 针对领域数据微调 embedding

#### 5.4 response 质量差

典型原因：

- prompt 模板差
- context 质量差
- 模型理解指令能力不足

对策：

- 迭代 prompt
- 用领域数据微调 LLM
- 换更合适的指令模型

#### 5.5 embedding 表示偏差大

常见原因：

- chunk 太长
- 模型本身弱
- 多语言未对齐

对策：

- 更小 chunk
- 更大 top-k
- 更好的多语言 embedding
- 领域微调 embedding

#### 5.6 prompt 变化导致效果波动

这是非常常见的真实问题。

对策：

- 固化模板
- 明确输出格式
- 建评测集
- 用自动化评估比较版本

#### 5.7 LLM 格式跟随能力不足

原文提到开源模型与 GPT 在格式服从性上的差异，这很典型。

可行方案：

- 更强的 instruction-tuned 模型
- 少样本示例
- 域内格式数据微调

### 6. 更高质量召回的常见手段

- 细粒度 chunking
- 关键词索引 + 原文映射
- ES + Faiss 混合检索
- query rewrite
- query expansion
- rerank
- metadata filtering
- 垂直领域 embedding 微调

## 十一、RAG 评测篇

### 1. 为什么要评测

因为 RAG 是链路系统，不仅要看最后答案，还要看检索是否正确。

### 2. 两类评测方式

#### 2.1 独立评测

分别评估：

- 检索模块
- 生成模块

检索常见指标：

- Hit Rate
- MRR
- NDCG
- Precision
- Recall

生成侧独立评估重点通常是：

- 上下文相关性
- 忠实度

#### 2.2 端到端评测

直接评估最终答案。

常见维度：

- 准确性
- 相关性
- 一致性
- 无害性
- 精确匹配 EM

### 3. RAG 的关键能力

原文提到的重点可以总结为：

- 抗噪声能力
- 拒答能力
- 信息综合能力
- 反事实稳健性

这是比单纯看“答对率”更高级的面试回答。

### 4. RAGAS

RAGAS 是较常用的自动评估框架。

核心看三类分数：

- answer faithfulness
- answer relevancy
- context relevancy

它常通过 LLM 把答案拆成陈述，再判断这些陈述是否被上下文支持。

### 5. ARES

ARES 也是自动化 RAG 评估框架，目标是用更少人工标注实现系统级评估。

特点：

- 用少量标注 + 合成数据
- 关注上下文相关性、答案忠实度、答案相关性
- 可结合置信区间做更稳健比较

### 6. 面试总结句

“RAG 的评估不能只看最终答案，至少要把检索质量、上下文质量和生成忠实度拆开看。”

## 十二、推理加速篇

### 1. 推理优化手段总览

原文分成三层：

- 算法层：蒸馏、量化
- 软件层：计算图优化、模型编译、内核融合
- 硬件层：更高效精度格式，例如 FP8

### 2. FasterTransformer

特点：

- 针对 Transformer encoder/decoder 做推理层优化
- 融合多种非 GEMM 操作
- 支持 FP16 / INT8 / FP8
- 处理 padding 优化

适合：

- NVIDIA 生态
- 高性能推理场景

### 3. TurboTransformers

特点：

- 支持 CPU/GPU
- smart batching
- 变长序列友好
- 对 LayerNorm 和 Softmax 做并行优化
- 关注内存管理

### 4. vLLM

这是当前最常见推理服务高频面试点。

#### 4.1 核心能力

- Continuous batching
- PagedAttention
- OpenAI API 兼容
- 高吞吐服务

#### 4.2 为什么快

核心原因不是“模型更好”，而是调度和内存管理更优：

- iteration-level 调度
- 更高效复用 KV Cache
- 降低显存碎片

#### 4.3 优点

- 吞吐高
- 服务能力强
- 工程使用方便

#### 4.4 缺点

- 自定义模型适配有门槛
- 适配器支持曾经较弱，工程兼容性要确认
- 某些量化和特殊结构支持不如专用框架完整

### 5. TGI

Text Generation Inference 是 Hugging Face 的推理服务框架。

特点：

- Docker 化方便
- 生态接 Hugging Face 模型方便
- 提供监控与服务能力
- 支持 flash attention、paged attention、张量并行、量化等能力

缺点：

- 深度定制门槛较高
- 文档和源码需要一起看
- 对适配器和特殊模型支持要具体确认

### 6. 什么时候选 vLLM，什么时候选 TGI

面试可这样答：

- 追求高吞吐 OpenAI 风格服务，优先考虑 vLLM
- 需要 Hugging Face 部署生态和现成服务能力，可考虑 TGI
- 最终还要看模型结构支持、量化需求、LoRA 支持和运维方式

## 十三、显存问题篇

### 1. 显存都花在哪

训练时主要包括：

- 参数
- 梯度
- 优化器状态
- 激活值
- 临时 buffer

推理时主要包括：

- 参数
- KV Cache
- 临时计算 buffer

### 2. 为什么训练显存远大于推理

因为训练不仅要前向，还要：

- 保留反向需要的中间结果
- 保存梯度
- 维护优化器状态

如果是 Adam，优化器状态通常很占显存。

### 3. 显存不够怎么办

常见手段：

- 混合精度
- 梯度检查点
- ZeRO / FSDP
- TP / PP
- CPU / NVMe offload
- LoRA / QLoRA
- 量化推理

### 4. 如何估算模型显存

面试不一定要求死公式，但要会定性分析：

- 参数量越大，基础占用越大
- 序列越长，KV Cache 和激活占用越高
- batch 越大，显存增长越明显
- 训练精度越高，显存压力越大

### 5. 如何评估 GPU 利用率

不能只看利用率百分比，还要看：

- 实际 TFLOPS
- 显存带宽利用
- step time
- 通信时间占比
- 数据读取是否阻塞

### 6. 如何查看训练瓶颈

原文提到的工具思路包括：

- DeepSpeed 报告
- torch profiler
- 通信开销分析
- FLOPS 统计

面试里可以说：

“先分清是算力瓶颈、显存瓶颈还是通信瓶颈，再做优化。”

## 十四、增量预训练篇

### 1. 为什么做增量预训练

主要目的：

- 注入新领域知识
- 适配特定语料风格
- 在原模型基础上持续训练

它和 SFT 的区别是：

- 增量预训练更偏“继续学语言和领域知识”
- SFT 更偏“学任务格式和指令跟随”

### 2. 做前需要准备什么

- 明确领域目标
- 清洗高质量语料
- 去重
- 保证语料分布合理
- 确认 tokenizer 是否匹配
- 规划训练预算和评估集

### 3. 训练框架

通常仍会用：

- DeepSpeed
- Megatron-LM
- Transformers + FSDP

取决于模型规模和资源。

### 4. 一般流程

1. 准备语料
2. 分词与打包
3. 配置学习率和 warmup
4. 启动继续预训练
5. 监控 loss 和下游验证集
6. 决定是否继续、回退或切换阶段

### 5. 数据量怎么定

没有绝对值，要看：

- 模型大小
- 领域偏移程度
- 目标是轻度适配还是深度迁移

原则：

- 数据太少，效果不稳定
- 数据太杂，容易稀释目标能力

### 6. loss 上升正常吗

有时候正常，有时候不正常。

要结合判断：

- 是否切到了分布差异极大的新语料
- 学习率是否过大
- warmup 是否过短
- 是否发生灾难性遗忘或训练不稳定

### 7. 学习率和 warmup

面试要点：

- 继续预训练通常比从头预训练更谨慎
- 学习率往往更小
- warmup 仍然重要
- rewarmup 可能影响迁移稳定性

### 8. 继续预训练的主要风险

- 过拟合新领域
- 灾难性遗忘
- 训练成本高
- 下游收益不一定立刻体现

## 十五、蒸馏、量化与压缩篇

### 1. 知识蒸馏是什么

知识蒸馏是让小模型学习大模型输出分布、隐藏表示或中间知识的过程。

目标：

- 降低部署成本
- 保留尽可能多的效果

### 2. 蒸馏方式

常见包括：

- logit 蒸馏
- feature 蒸馏
- relation 蒸馏
- response 蒸馏

在 LLM 里也常见：

- teacher 生成数据
- student 模仿 teacher 回答

### 3. 量化是什么

量化是把权重、激活或 KV Cache 从高精度表示压缩到低精度表示。

常见形式：

- INT8
- INT4
- FP8

优点：

- 降低显存
- 降低带宽开销
- 提升部署效率

风险：

- 精度损失
- 某些模型对量化更敏感

### 4. 模型压缩和加速的方法

原文这部分可以归纳为：

- 蒸馏
- 量化
- 剪枝
- 结构重设计
- kernel 优化
- 图编译
- 并行化推理

### 5. 面试里怎么谈取舍

不要只说“模型变小了”，而要说：

- 节省的是显存还是时延
- 影响的是首 token 延迟还是吞吐
- 损失的是通用能力还是细粒度能力
- 是否增加了工程复杂度

## 十六、面试高频问法速答

### 1. 为什么不用微调代替 RAG

- 微调适合学风格和能力
- RAG 适合外挂可更新知识
- 两者并不互斥

### 2. 为什么 TP 通常要求 NVLink

- 因为 TP 通信频繁
- 慢网络会让通信成本吞掉计算收益

### 3. 为什么 MQA 能提升推理速度

- 共享 K/V
- KV Cache 更小
- 显存访问更少

### 4. 为什么 FlashAttention 更快

- 不是数学公式变了
- 是访存模式更优
- HBM 访问减少，SRAM 利用更高

### 5. 为什么 chunk 不能一味做大

- 会引入噪声
- 增加幻觉风险
- 降低检索精度

### 6. RAG 效果差先查哪里

优先级通常是：

1. 文档解析和切分
2. embedding 质量
3. 检索策略
4. rerank
5. prompt
6. LLM 本身

### 7. 为什么分类常用交叉熵

- 概率建模更匹配
- 梯度性质更好
- 对错误高置信预测惩罚更强

## 十七、复习建议

如果你是为面试准备，这份文档建议按下面顺序看：

1. Tokenizer、Transformer 基础组件、Attention 优化
2. 并行训练、显存、推理加速
3. RAG 基础、优化、评测
4. 增量预训练、蒸馏、量化

回答时尽量使用统一结构：

1. 先定义问题
2. 再讲核心机制
3. 再讲优缺点
4. 最后落到工程实践

一句话总结：

大模型面试真正考的不是你能背多少术语，而是你能不能把模型原理、系统瓶颈和工程取舍串成一条完整链路。
