## 项目

## 项目一：智能问答

### 目标
- 上传 PDF / Word / 网页
- 自然语言提问，返回答案 + 引用来源
- 支持多文档检索

### 技术栈
```
文档解析：Apache PDFBox + Tika
分块：LangChain4j DocumentSplitters（500 tokens, overlap 50）
Embedding：text-embedding-3-small
向量库：Qdrant（Docker 本地）
生成：claude-sonnet-4-6 或 gpt-4o
框架：Spring Boot 3 + LangChain4j
```

### 核心流程
```java
// 入库
EmbeddingStoreIngestor.builder()
    .documentSplitter(DocumentSplitters.recursive(500, 50))
    .embeddingModel(embeddingModel)
    .embeddingStore(embeddingStore)
    .ingest(documents);

// 查询
ContentRetriever retriever = EmbeddingStoreContentRetriever.builder()
    .embeddingStore(embeddingStore)
    .embeddingModel(embeddingModel)
    .maxResults(5)
    .minScore(0.65)
    .build();
```

### 关键设计决策

**为什么用 Qdrant 而不是 FAISS**：Qdrant 有完整的 HTTP API，支持元数据过滤，生产可用。FAISS 是内存库，不适合多文档场景。

**为什么 chunk size 选 500**：FAQs 通常是一问一答的段落结构，500 tokens 能包含完整的问题和回答，不会切断语义。

**如何返回来源**：存储文档名、页码到 Qdrant 元数据，检索结果中携带这些元数据，拼入 Prompt 要求模型引用。

### 评估指标
- 问题命中率（能否找到答案）> 85%
- Faithfulness > 4.0 / 5.0
- 用户满意度（点赞/踩）

## 项目二：ChatBI / Text2SQL

### 目标
- 用自然语言提问（"上个月销售额最高的 TOP 10 产品是什么"）
- 系统生成 SQL 并查询数据库
- 返回表格数据 + 自然语言结论

### 技术栈
```
SQL 生成：claude-sonnet-4-6（代码能力强）
数据库：MySQL / PostgreSQL
安全层：白名单 SQL 解析
可视化：ECharts（图表生成）
框架：Spring Boot 3
```

### 核心 Prompt 设计

```
你是一个 SQL 专家。根据用户的自然语言问题生成 SQL 查询。

数据库表结构：
{table_schema}

Few-shot 示例：
用户：上个月销售额 TOP 5 的产品
SQL：SELECT product_name, SUM(amount) as total_sales 
     FROM orders 
     WHERE created_at >= DATE_SUB(NOW(), INTERVAL 1 MONTH)
     GROUP BY product_name 
     ORDER BY total_sales DESC 
     LIMIT 5;

规则：
1. 只生成 SELECT 语句
2. 必须包含 LIMIT（默认 LIMIT 100）
3. 如果无法生成，返回 {"error": "原因"}

用户问题：{question}
SQL：
```

### 关键设计：Schema 注入

**不能把整个数据库 Schema 都注入**（token 太多，模型效果差）：

```java
// 动态 Schema 注入：只注入与问题相关的表
public String buildPrompt(String question) {
    // 用 embedding 找最相关的表
    List<TableInfo> relevantTables = tableSchemaRetriever.findRelevant(question, topK=5);
    
    String schema = relevantTables.stream()
        .map(t -> t.getCreateStatement() + "\n-- 字段说明：" + t.getColumnDescriptions())
        .collect(Collectors.joining("\n\n"));
    
    return PROMPT_TEMPLATE.formatted(schema, question);
}
```

### SQL 安全层

```java
public QueryResult executeSafeSql(String sql) {
    // 1. 只允许 SELECT
    if (!sql.trim().toUpperCase().startsWith("SELECT")) {
        throw new SecurityException("只允许查询操作");
    }
    // 2. 强制 LIMIT
    if (!sql.toUpperCase().contains("LIMIT")) {
        sql += " LIMIT 100";
    }
    // 3. 超时控制
    return jdbcTemplate.execute(connection -> {
        connection.setQueryTimeout(30);
        return connection.prepareStatement(sql).executeQuery();
    });
}
```

### 项目亮点
- "为什么要动态 Schema 注入"：减少 token 消耗，提高生成准确率
- "SQL 注入如何防护"：白名单 + 只允许 SELECT + PreparedStatement
- "如何处理模型生成 SQL 有语法错误"：捕获异常后把错误信息回填给 LLM，让它自动修复

## 项目三：企业知识库助手

### 目标
- 多数据源接入（内部文档、Confluence、飞书）
- 权限隔离（不同用户看不同文档）
- 多轮对话 + 来源追溯
- 完整的评估和监控

### 技术栈
```
框架：Spring Boot 3 + Spring AI
向量库：Qdrant（权限过滤）
检索：混合检索（向量 + Elasticsearch BM25）
重排：bge-reranker-large
可观测：LangFuse
评估：Ragas（Python 脚本独立运行）
缓存：Redis（语义缓存 + 会话管理）
```

### 架构图

```
用户
  ↓ HTTP/WebSocket
API Gateway（鉴权、限流）
  ↓
RAG Service
  ├── Session Manager（Redis，多轮历史）
  ├── Query Rewriter（LLM）
  ├── Hybrid Retriever
  │   ├── Vector Search（Qdrant + 权限过滤）
  │   └── BM25 Search（Elasticsearch）
  ├── Reranker（bge-reranker-large）
  └── Generator（Claude Sonnet）
  ↓
LangFuse（可观测）
```

### 关键设计：多轮对话历史管理

```java
// 超过 20 轮后压缩历史
public List<ChatMessage> getCompressedHistory(String sessionId) {
    List<ChatMessage> history = chatMemory.getMessages(sessionId);
    
    if (history.size() > 20) {
        String summary = llm.summarize(history.subList(0, 15));
        List<ChatMessage> compressed = new ArrayList<>();
        compressed.add(new SystemMessage("对话历史摘要：" + summary));
        compressed.addAll(history.subList(15, history.size()));
        return compressed;
    }
    return history;
}
```

### 核心设计
- chunk size 选 512 的原因（实验对比过 256、512、1024）
- top-K 设为 5 的原因（5 个 chunk ≈ 2000 tokens，留足生成空间）
- 语义缓存阈值 0.92（通过对评估集 false positive 率调优）
- Faithfulness 基线：0 跑完评估集后，Prompt 迭代提升到 4.2+

## 项目四：Agent运营助手

### 目标
- 接受自然语言指令（"给我看看上周新注册但没完成首购的用户，发一封召回邮件"）
- 自动完成：查数据库 → 分析数据 → 生成个性化内容 → 发送邮件
- 可追踪每一步操作

### 工具集设计

```java
public class MarketingTools {

    @Tool("查询满足条件的用户列表。支持条件：注册时间、是否购买、城市等")
    public List<User> queryUsers(@P("SQL WHERE 条件") String condition) {
        // 安全校验：只允许 SELECT users 表
        return userRepository.findByCondition(condition, 1000);
    }

    @Tool("分析用户列表的统计信息，返回人数、城市分布、设备分布等")
    public UserStats analyzeUsers(@P("用户ID列表") List<Long> userIds) {
        return analyticsService.analyze(userIds);
    }

    @Tool("根据用户信息生成个性化邮件内容。需要提供用户特征和邮件主题")
    public String generateEmailContent(
        @P("用户特征描述") String userProfile,
        @P("邮件主题") String subject
    ) {
        // 调用 LLM 生成个性化文案
        return contentLlm.generate(userProfile, subject);
    }

    @Tool("发送邮件给指定用户列表。重要：调用此工具前必须先确认用户数量和内容")
    public SendResult sendEmail(
        @P("用户ID列表") List<Long> userIds,
        @P("邮件主题") String subject,
        @P("邮件内容模板") String template
    ) {
        // 防护：限制单次发送数量
        if (userIds.size() > 10000) {
            return SendResult.error("单次发送不能超过1万人，请分批操作");
        }
        return emailService.send(userIds, subject, template);
    }
}
```

### 关键设计：危险操作的人工确认

```java
// Agent 工具链中，发送邮件前强制二次确认
@Tool("确认并执行批量邮件发送。仅当用户明确说'确认发送'后才调用此工具")
public String confirmAndSend(@P("发送计划JSON") String plan) {
    // 记录审计日志
    auditLog.record(currentUser, "SEND_EMAIL", plan);
    return emailService.executePlan(plan);
}
```

### 项目亮点
- "为什么 Agent 不直接让模型决定发送"：高风险操作（发邮件、扣款）必须有人工确认节点，不能完全依赖模型判断
- "如何追踪 Agent 每一步"：每次工具调用记录到 LangFuse Trace，包括输入、输出、耗时
- "Agent 失败怎么办"：每步保存状态到 Redis，失败后可从断点恢复，不需要从头开始

## 通用面试技巧

**展示项目时的结构**：
1. 业务场景（为什么做）
2. 技术选型理由（为什么用这个技术，不用那个）
3. 遇到的真实问题和解法（最有说服力的部分）
4. 量化指标（Faithfulness 从 X 提升到 Y，成本降低 Z%）

**有数字远比泛泛而谈更有说服力**：
- "Reranker 让 Faithfulness 从 3.1 提升到 4.3"
- "语义缓存让 60% 的重复问题直接命中，成本降低 40%"
- "混合检索比纯向量检索的 Context Recall 提升了 18%"
