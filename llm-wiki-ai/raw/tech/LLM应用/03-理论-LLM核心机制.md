## 一、Tokenizer：被低估的基础设施

Tokenizer 不是简单的"预处理"，它直接决定：

- **上下文利用率**：中文 BPE 效率低时，同等 token 预算表达的信息量更少
- **推理成本**：同等内容产生的 token 数不同，API 费用不同
- **OOV 处理**：字符级回退 vs 未知 token
- **跨语言能力**：词表的多语言覆盖程度

### 三大主流方案

| 方案 | 代表模型 | 核心思路 |
|------|----------|----------|
| BPE | GPT 系列、LLaMA | 频率驱动的字节对合并 |
| WordPiece | BERT | 最大化语言模型概率的子词合并 |
| SentencePiece | LLaMA、T5、ChatGLM | 语言无关，把空格也当普通符号，可基于 BPE 或 Unigram LM |

**BPE 流程**：字符级初始化 → 统计高频相邻对 → 每轮合并最常见对 → 重复到目标词表大小。
**WordPiece vs BPE**：合并标准不同，WordPiece 更偏语言模型概率提升，而非纯频率。

**中文分词问题**：LLaMA 原始词表对中文不友好，平均一个汉字可能拆成多个 token，导致上下文窗口浪费。扩展中文词表（如 Chinese-LLaMA）可显著提升中文编码效率。

## 二、大模型训练：显存、吞吐、通信的三角平衡

### 2.1 为什么单卡放不下

7B 参数模型（FP16）：
- 参数：7B × 2 Bytes = **14 GB**
- 梯度：14 GB
- Adam 优化器状态：56 GB（一阶矩 + 二阶矩 = 4× 参数量）
- 激活值：依 batch size 和序列长度线性增长

**全精度训练总显存约 84+ GB**，远超单张 A100（80 GB）。

### 2.2 四种并行策略

| 策略 | 分割维度 | 通信开销 | 适合场景 |
|------|----------|----------|----------|
| 数据并行（DP） | Batch | 梯度同步（AllReduce） | 模型能放进单卡 |
| 张量并行（TP） | 矩阵列/行 | 高频 AllReduce | 单层太大 |
| 流水线并行（PP） | 层 | 低频，有 bubble | 层数多 |
| 专家并行（EP） | MoE 专家 | 稀疏通信 | MoE 架构 |

**核心矛盾**：大模型训练不是"能不能跑"，而是**显存、吞吐、通信三者如何平衡**。

### 2.3 ZeRO：显存优化的核心

ZeRO（Zero Redundancy Optimizer）把模型状态分片到多卡：

| Stage | 分片内容 | 显存节省 | 通信代价 |
|-------|----------|----------|----------|
| ZeRO-1 | 优化器状态 | 4x（Adam 下） | 低 |
| ZeRO-2 | 优化器状态 + 梯度 | 8x | 中 |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | 64x+ | 高 |

ZeRO-3 显存节省最大，但参数需要通信恢复，适合显存极度紧张时。

### 2.4 混合精度训练

- **FP16**：计算快，但数值范围窄，容易溢出，需 loss scaling
- **BF16**：数值范围与 FP32 相同，训练更稳定，现代大模型首选
- **FP32**：主副本保留精度，计算时转 FP16/BF16

## 三、推理加速：瓶颈在显存带宽，不在算力

### 3.1 KV Cache：自回归推理的核心优化

**为什么需要**：自回归生成每步都要重算所有历史 token 的 K、V，代价 O(n²)。KV Cache 将已计算的 K、V 缓存，每步只计算新 token，代价降为 O(n)。

**代价**：KV Cache 占用显存随序列长度线性增长，是长文本推理的显存主要瓶颈。

$$\text{KV Cache 显存} = 2 \times \text{层数} \times \text{头数} \times d_{\text{head}} \times \text{序列长度} \times \text{精度字节数}$$

### 3.2 PagedAttention（vLLM 核心）

**问题**：传统 KV Cache 预分配连续内存，不同长度请求导致严重内存碎片，显存利用率仅 20-40%。

**解法**：借鉴操作系统虚拟内存思想，将 KV Cache 分成固定大小的 Block 按需分配，Block 不必连续。

**效果**：显存利用率提升到 90%+，吞吐大幅提升。

### 3.3 Continuous Batching

**问题**：静态批处理要等最长序列完成才释放资源，GPU 利用率低。

**解法**：序列生成完成后立即插入新请求，动态填充 batch 槽位。

### 3.4 推理服务核心指标

| 指标 | 含义 | 影响 |
|------|------|------|
| TTFT（首 token 延迟） | 从请求到第一个 token 的时间 | 用户感知响应速度 |
| TBT（token 间延迟） | 逐 token 生成间隔 | 输出流速 |
| 吞吐（Tokens/s） | 系统并发处理能力 | 成本和容量规划 |

## 四、注意力机制优化变体

原始 Multi-Head Attention 存在两个问题：
1. KV Cache 随头数线性增长（每个头独立的 K、V）
2. 计算复杂度 O(n²)

### 4.1 KV Cache 缩减

| 方案 | 核心思路 | KV Cache 节省 | 质量影响 |
|------|----------|---------------|----------|
| MQA（多查询注意力） | 所有头共享一组 K、V | H 倍（头数倍） | 有损失 |
| GQA（分组查询注意力） | G 组头共享一组 K、V | H/G 倍 | MHA 和 MQA 的折中 |

LLaMA 2 70B、Mistral、Qwen 等现代模型广泛使用 GQA。

### 4.2 计算加速

| 方案 | 核心思路 | 收益 |
|------|----------|------|
| FlashAttention | IO 感知分块计算，避免频繁读写 HBM | 速度 2-4x，显存 O(n) |
| 稀疏注意力 | 只计算局部或特定模式位置 | 长序列计算量降低 |
| 线性注意力 | 核函数近似，复杂度 O(n) | 精度有损失 |

**FlashAttention 核心原理**：标准 Attention 需要将 n×n 矩阵写入 HBM（高带宽内存）再读回，IO 是瓶颈。FlashAttention 分块计算，每块 fit 进 SRAM（片上内存），大幅减少 HBM 访问次数。

## 五、显存优化技术全景

| 技术 | 节省的显存 | 代价 |
|------|-----------|------|
| 混合精度（FP16/BF16） | 参数 + 激活 | BF16 数值范围够用，FP16 需 loss scaling |
| 梯度检查点 | 激活显存（约 √n 倍） | 重计算约增加 30% 训练时间 |
| ZeRO Stage 3 | 参数 + 梯度 + 优化器状态 | 通信量增大 |
| LoRA | 可训练参数量（减少 >95%） | 表达能力略有限制 |
| INT8/INT4 量化 | 参数显存减半/四分之一 | 精度损失，需校准 |
| QLoRA | LoRA + NF4 量化基座 | 训练速度略慢 |

**核心原则**：显存优化是手段，**以可接受的精度或速度损失换取更低的部署成本**才是目标。

## 六、参数高效微调（PEFT）

### 6.1 LoRA：低秩适配

**核心假设**：模型权重在微调时的更新矩阵是低秩的。

$$W' = W + \Delta W = W + AB, \quad A \in \mathbb{R}^{d \times r}, B \in \mathbb{R}^{r \times k}$$

其中 r 远小于 d 和 k，可训练参数量减少 99%+。推理时直接合并 AB 到原权重，不增加推理延迟。

**超参数**：
- rank r：越大表达能力越强，一般 4-64
- alpha：缩放系数，通常 = 2r
- 作用于哪些层：通常 Q、V 矩阵，有时加 K、FFN

### 6.2 QLoRA

4-bit 量化基座模型 + LoRA 微调，使单卡 48GB 显存可微调 65B 参数模型。

关键技术：NF4（Normal Float 4bit）量化、双重量化、分页优化器。

### 6.3 为什么优先 RAG 而不是微调

| | RAG | 微调 |
|---|---|---|
| 知识更新成本 | 低（更新文档库） | 高（重训练） |
| 知识溯源 | 可以（返回来源文档） | 不可以（知识被编码进参数） |
| 适用场景 | 知识更新频繁 | 行为/风格/格式对齐 |

**工程结论**：先用 RAG，RAG 搞不定的行为一致性问题再考虑微调。

## 七、量化

### 7.1 量化基础

将浮点参数（FP32/FP16）映射到低精度整数（INT8/INT4）：

$$x_q = \text{clamp}(\text{round}(x/s + z), -2^{b-1}, 2^{b-1}-1)$$

其中 s 是缩放因子，z 是零点偏移，b 是位宽。

### 7.2 主要方案

| 方案 | 时机 | 特点 |
|------|------|------|
| PTQ（训练后量化） | 推理前 | 简单快速，需要校准集 |
| QAT（量化感知训练） | 训练中 | 效果最好，成本高 |
| GPTQ | 训练后，逐层 | 4bit 精度较高 |
| AWQ | 训练后，激活感知 | 保护重要权重，精度好 |

**为什么量化需要校准集**：量化会改变数值分布，需要用少量真实数据校准缩放因子，否则精度损失严重。

## 八、MoE（混合专家）架构

**核心思想**：不是每次前向都激活全部参数，而是通过路由网络（Router）选择少数专家（Expert FFN）激活。

```
输入 token → Router（Softmax） → 选 Top-k 专家 → 仅这 k 个 FFN 计算 → 加权合并
```

**优势**：参数总量大（更强知识存储），但每次激活参数少（推理成本可控）。
**代价**：负载均衡问题（某些专家可能被过度选择），训练稳定性更难控制。

代表模型：Mixtral 8×7B（8 个专家，每次激活 2 个）、DeepSeek-MoE、GPT-4（推测）。

## 九、RLHF 与对齐

### 9.1 SFT → RM → PPO 流程

1. **SFT（监督微调）**：用高质量指令-回答对微调，教模型"怎么回答"
2. **RM（奖励模型）**：训练一个模型评估回答质量（人类排序反馈）
3. **PPO（强化学习）**：用 RM 作为奖励信号，进一步优化 LLM 输出分布

### 9.2 DPO：直接偏好优化

绕过 RM 和 PPO，直接用偏好对（好答案 vs 差答案）优化策略模型，训练更稳定，效果接近 RLHF。

### 9.3 GRPO（DeepSeek 使用）

Group Relative Policy Optimization：去掉 value model，通过组内相对排序估计优势，更高效。

## 十、大模型缩放规律（Scaling Laws）

**Chinchilla 缩放规律**：给定计算预算，最优策略是**模型参数和训练 token 数等比增长**（而非只增大模型）。

- 训练 1T token 的 7B 模型 ≈ 训练 100B token 的 70B 模型（计算量相近，但前者推理成本更低）
- 实践启示：过去很多模型是"undertrained"的（参数多但训练 token 少）
