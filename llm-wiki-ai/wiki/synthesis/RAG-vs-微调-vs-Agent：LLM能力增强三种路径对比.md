# RAG vs 微调 vs Agent：LLM能力增强三种路径对比

> 当 LLM 的基础能力不满足需求时，有三条主要增强路径：RAG（注入外部知识）、微调（改变模型行为）、Agent（扩展行动能力）。三者不是竞争关系，而是针对不同问题类型的互补工具。

## 核心论点

**选择哪条路径，本质上取决于"是什么限制了模型"**：

- 模型回答问题时**缺乏特定领域知识**或知识已过时 → 用 **RAG**
- 模型的**输出风格、格式或特定行为**不符合要求 → 用**微调（LoRA）**
- 模型需要**与外部系统交互、执行多步操作** → 用 **Agent**

错误的选择：用微调试图"注入知识"（效果差、成本高），用 RAG 试图改变模型"说话方式"（治标不治本），用 Agent 处理单次查询（过度工程化）。

---

## 三种路径的核心对比

| 维度 | RAG | 微调（LoRA） | Agent |
|------|-----|-------------|-------|
| **解决的核心问题** | 知识截止、幻觉、需要引用来源 | 风格/格式/行为定制，领域术语理解 | 需要实时数据、多步操作、外部系统交互 |
| **知识更新方式** | 更新向量数据库（分钟级） | 重新训练（小时-天级） | 工具实时调用（秒级） |
| **实现复杂度** | 中等（分块/检索/Reranker 管道） | 中等（需要训练数据和 GPU） | 高（工具描述/失败处理/状态管理） |
| **可解释性** | 高（可显示引用来源） | 低（行为变化原因不透明） | 中（工具调用链可追踪） |
| **幻觉风险** | 低（有文档支撑） | 中（仍可能虚构） | 中（取决于工具返回质量） |
| **延迟** | 中（检索 + Reranker 增加 200-500ms） | 无额外延迟（合并后） | 高（多步推理，每步 LLM 调用） |
| **适合场景** | 知识库问答、文档检索 | 风格一致性、格式规范、角色扮演 | 数据查询+计算+操作的复合任务 |

---

## 何时选择 RAG

**明确信号**：用户问的是"XX 产品的规格是什么""XX 规定是怎么说的"，答案存在于文档中但 LLM 不知道或可能幻觉。

**RAG 的关键决策链**：
1. 文档质量 → 解析是否准确（MinerU 处理复杂 PDF）
2. 分块策略 → 针对文档类型调整（FAQ 512-800T，技术文档 200-400T）
3. 检索方式 → 混合检索（向量 + BM25 + RRF）优于纯向量
4. 精排 → Reranker 是 ROI 最高的单点优化（Faithfulness 从 3.1→4.3）
5. 评估 → RAGAS 四指标持续监控

**RAG 不适合的情况**：需要推理/计算（RAG 只能检索，不能推理），实时数据（RAG 基于离线索引），需要改变模型行为（应用微调）。

---

## 何时选择微调（LoRA）

**明确信号**：模型回答的内容大体正确，但格式、风格、专业术语不对；或者需要模型学习大量领域特定的语言模式（医疗、法律、代码风格）。

**微调适合的具体场景**：
- 输出格式严格化（特定 JSON schema、报告格式）
- 风格定制（品牌语气、特定说话风格）
- 专业术语理解（领域缩写、内部命名规范）
- 减少幻觉（领域知识有限的模型需要领域微调以减少乱答）

**微调不适合的情况**：用微调注入最新知识（知识截止问题 RAG 更合适），数据量 < 1000 条（容易过拟合，LoRA 的低秩约束有一定帮助），知识更新频繁（重新训练成本高）。

**LoRA 的工程要点**：rank=4-64（越大表达能力越强但越容易过拟合），学习率比预训练小 10-100 倍，对 Q/K/V/O 矩阵应用 LoRA，QLoRA 可在单张 A100 上微调 70B 模型。

---

## 何时选择 Agent

**明确信号**：单次 LLM 调用无法完成任务，需要查询数据库 + 计算 + 生成报告 + 发送邮件的复合流程；或者需要实时数据（股价、天气、最新新闻）。

**Agent 的设计优先级**：
1. **单 Agent 优先**：工具 ≤ 10 个、步骤 ≤ 5 时，单 Agent 足够；多 Agent 引入协调成本
2. **工具描述是第一杠杆**：做什么 + 何时调用 + 参数格式 + 返回内容，四要素完整
3. **失败处理比规划重要**：工具返回结构化错误信息，90% 问题在这里
4. **危险操作二次确认**：发邮件/支付必须有独立校验 + 人工确认节点
5. **状态可追踪**：LangFuse 记录完整 trace，不可追踪 = 不可调试

---

## 组合使用的最佳实践

三者可以且经常应该组合：

**RAG + Agent**（最常见组合）：RAG 作为 Agent 的一个知识检索工具，Agent 负责多步推理和操作；用户问"基于我们的产品手册，帮我起草一封给客户的技术支持邮件并发送"，Agent 调用 RAG 工具（检索手册）→ 生成邮件 → 调用发送工具（需人工确认）。

**RAG + 微调**：对领域专用模型先微调（改善术语理解和输出格式），再用 RAG 注入实时知识，两者互补——微调改善"说话方式"，RAG 保证"知识准确"。

**微调 + Agent**：微调定制模型的工具调用格式理解（特别是对私有 API 的函数调用），Agent 负责多步推理；在需要严格格式的企业 Agent 场景有价值。

## 决策树

```
需要 LLM 增强能力？
├── 主要问题是"不知道"或"知识过时"？
│   └── 是 → RAG（检索外部知识库）
│       ├── 知识更新频繁？ → 确保向量数据库支持增量更新
│       └── 需要引用来源？ → RAG 天然支持
├── 主要问题是"说话方式不对"或"格式不对"？
│   └── 是 → 微调（LoRA）
│       ├── 数据量 < 1000？ → 考虑 Few-shot Prompt 替代
│       └── 知识更新频繁？ → 微调不合适，改用 RAG
└── 需要执行操作或获取实时数据？
    └── 是 → Agent（工具调用）
        ├── 操作步骤 ≤ 5？ → 单 Agent
        └── 操作步骤 > 5 或需要专业子任务？ → 多 Agent（先证明单 Agent 不够）
```
