# 策略：AI 高效使用技巧全景

> 核心观点：高效使用 AI 不是找到"神奇咒语"，而是理解模型的工作方式，系统化地设计输入条件。

---

## 第一性原理：你在做什么

模型输出什么，由它当前接收到的上下文完全决定。"提示词"本质上是在**收缩模型的输出概率空间**，让它更难产生你不想要的结果。

所以：
- 不是"命令模型"，是"缩小它的选择范围"
- 不是"写好看的句子"，是"减少歧义、补齐信息、限制边界"
- 不是"找魔法咒语"，是"系统化设计输入条件"

---

## 一、基础技巧：让模型理解你要什么

### 1.1 角色 + 任务 + 约束三件套

最小有效 Prompt 结构：
```
[你是谁/什么角色] + [做什么任务] + [输出要求/约束]
```

示例（差）：
> 帮我写一封邮件

示例（好）：
> 你是一位专业的商务写作顾问。帮我写一封向客户道歉的邮件，客户因为我们系统延迟损失了业务机会。要求：语气诚恳但不过度卑微，300字以内，提出具体补偿方案占位符。

**关键**：任务、受众、格式、长度约束——缺一个结果就会漂移。

### 1.2 示例优于规则（Few-shot）

与其写大量规则说明，不如给 2-3 个正确示例。

```
以下是我想要的输出格式：

输入：苹果
输出：水果类 | 常见 | 可生食

输入：龙葵
输出：植物类 | 少见 | 有毒

现在处理：枸杞
```

**原因**：规则需要模型解释后执行，示例直接展示模式，误解空间更小。参见 [[方法-Few-Shot-Prompting]]。

### 1.3 明确告诉模型"不要什么"

不要假设模型能猜到你的隐含排除项：
- "不要使用技术术语"
- "不要给出多于3个选项"
- "如果信息不足，直接说不知道，不要猜测"

## 二、进阶技巧：提升复杂任务的质量

### 2.1 让模型先思考再回答（Chain-of-Thought）

复杂推理任务，先要求模型列出步骤再给答案：

```
请先分析这个法律条款的关键要素，列出你的推理过程，再给出最终解释。
```

或直接用："让我们一步步思考"触发 CoT 模式。

**适合**：数学计算、逻辑推理、多条件判断、代码调试。
**不适合**：简单查询、格式转换（多余的思考步骤反而引入噪音）。参见 [[方法-Chain-of-Thought]]。

### 2.2 首尾位置放最重要的内容

实验一致显示：**模型对 Prompt 首部和尾部注意力最强，中间部分容易被稀释**（Lost in the Middle 效应）。

实践：
- 关键约束放在最前或最后
- 长文档中，重要参考信息放在 Prompt 结尾（离生成点最近）
- 避免把核心指令埋在大段背景信息中间

### 2.3 用分隔符明确结构

大模型对结构化输入理解更稳定：

```
<任务>
总结以下会议记录，提取3个行动项
</任务>

<会议记录>
[内容...]
</会议记录>

<输出格式>
行动项（负责人 | 截止日期）：
1.
2.
3.
</输出格式>
```

Claude 尤其推荐 XML 标签；Gemini 更倾向简洁直接的文本结构。

### 2.4 迭代而不是一次写完美

对于复杂任务：
1. 先用简单 Prompt 跑一次，看模型理解偏差在哪
2. 针对具体偏差加约束，而不是重写整个 Prompt
3. 固定好的 Prompt 后，变化的是输入数据

## 三、模型差异：Claude vs Gemini 的使用策略

### Claude（Anthropic）
- **字面执行**：会尽量完全遵守你的每一条指令，包括你没想到的边界情况
- **XML 标签**：显式结构对 Claude 效果更好（`<instructions>`, `<context>`, `<output>`）
- **长上下文**：适合处理大量文档，多文档对比分析表现好
- **思考预算**：复杂任务可以用 extended thinking，给它更多推理空间
- **注意**：约束越清晰越好，歧义指令 Claude 会按字面最保守解释

### Gemini（Google）
- **简洁直接**：提示更短更直接效果往往更好，不需要过多修饰
- **工具化**：在 API 和函数调用场景设计偏工具化使用
- **多模态**：图像+文本混合任务是强项
- **注意**：不需要像 Claude 那样写大量 XML 结构，保持简洁

参见：[[产品-Claude]] / [[产品-Gemini]] / [[组织-Anthropic]] / [[组织-Google]]

## 四、高频场景的最佳套路

### 场景1：信息提取/总结

```
从以下文本中提取：
- 关键人物（姓名 + 角色）
- 核心事件（时间 + 事件）
- 行动项（负责人 + 截止）

以结构化列表输出，没有信息的字段填"无"。

文本：[...]
```

### 场景2：代码生成/调试

- 提供**完整上下文**：语言版本、框架、已有代码
- 描述**预期行为 vs 实际行为**，不只是"不工作"
- 要求模型**解释修改原因**，不只给结果
- 复杂需求先要求**设计方案**再要求实现

### 场景3：写作/润色

- 明确**受众**（技术人员/普通用户/高管）
- 明确**目的**（说服/告知/激励）
- 给**参考样本**（"像这篇文章的风格"）
- 要求保留**具体数据和案例**，不接受空洞表述

### 场景4：分析/决策

- 先要求**列出分析框架**再填充内容
- 明确要求**列出反对意见**（避免模型只说你想听的）
- 要求**按重要性排序**，不要平铺所有点

## 五、避免常见陷阱

| 陷阱 | 症状 | 修复 |
|------|------|------|
| 过度修辞 | "请你作为一个极其专业的...大师..." | 去掉修辞，直接说任务 |
| 约束缺失 | 每次输出格式不同 | 加格式示例或明确结构要求 |
| 信息缺失 | 模型一直问问题或乱猜 | 补充必要背景，或明确说"不知道的填未知" |
| 任务太大 | 长输出质量快速下降 | 拆分为多个子任务分步完成 |
| 验证缺位 | 接受模型输出不检查 | 对重要输出做人工抽查，尤其是数字和引用 |

## 六、5C 框架：系统化提示词结构

对于需要重复使用的高质量 Prompt，用 5C 框架做结构化设计：

- **Character**：角色定义（你是谁）
- **Cause**：任务目标（做什么，为什么）
- **Constraint**：约束条件（格式、长度、风格、禁止项）
- **Contingency**：失败处理（信息不足时怎么做）
- **Calibration**：校准示例（期望输出的样本）

参见 [[框架-5C框架]] / [[主题-Prompt工程实践]]

## 七、从使用到工程化的边界

当你发现自己在做这些事时，说明已经需要工程化了：

- 每次手动粘贴大量背景文档 → 需要 [[概念-RAG]]
- 需要 AI 执行多个步骤/调用工具 → 需要 [[概念-Agent]]
- Prompt 经常变动，需要版本管理 → 需要可观测性（[[工具-LangFuse]]）
- 多人共用同一套 Prompt → 需要系统化 Prompt 管理

参见 [[策略-LLM应用开发入门路线]]
